# 웹 애플리케이션 디자인 — 플랫폼 개요

> 이 문서는 웹 앱 디자인 작업 시 **반드시 먼저 읽는 진입점**이다. 세부 가이드로 바로 가지 말고, 여기서 방향을 잡은 뒤 필요한 문서를 순서대로 참조하라.

---

## 웹 앱이란

브라우저 기반의 인터랙티브 도구. 사용자가 데이터를 조회·생성·수정·삭제하며, 세션 단위로 작업한다. 대시보드, 어드민, SaaS, 내부 툴, 에디터 등이 해당된다.

---

## 설계 워크플로우 — 이 순서를 따르라

### Step 1. 패러다임 선택 (필수, 최우선)

`web-app-paradigms.md`를 읽고 **화면의 지배적 패러다임을 하나 선택하라.**

| 패러다임 | 핵심 전제 | 대표 사례 |
|---------|----------|----------|
| **구조-주도** | 화면 = 작업 단위. 사이드바 + 콘텐츠 | 대시보드, 어드민, ERP |
| **커맨드-주도** | ⌘K 팔레트 + 키보드 중심. UI는 보조 | Linear, Raycast, VS Code |
| **캔버스-주도** | 무한 캔버스에서 직접 조작 (드래그/리사이즈/스냅) | Figma, Miro, Obsidian |
| **피드-주도** | 시간순 스트림. 읽음/안읽음, 고정/북마크 | Twitter, Slack, GitHub Activity |

> **판단 기준**: "사용자가 이 화면에서 주로 하는 행위가 무엇인가?"
> - 구조화된 CRUD → 구조-주도
> - 빠른 실행과 전환 → 커맨드-주도
> - 자유 배치와 공간 조작 → 캔버스-주도
> - 최신 정보 소비와 반응 → 피드-주도

하나의 제품 안에서도 화면마다 패러다임이 다를 수 있다. **하지만 한 화면에 두 패러다임을 섞지 말라.**

### Step 2. 컴포넌트 패턴 적용 (구현 시)

`web-app-components.md`를 읽고 화면에 필요한 컴포넌트의 설계 규칙을 따르라.

| 컴포넌트 | 언제 참조하는가 |
|---------|---------------|
| 데이터 테이블 | 목록/그리드 형태의 데이터 표시 시 |
| 폼 | 사용자 입력을 받는 모든 화면 |
| 빈 상태 | 데이터가 없는 상태를 다룰 때 |
| 로딩 상태 | 비동기 데이터 로드가 있을 때 |
| 알림/토스트 | 사용자 피드백 시스템 설계 시 |
| 모달/다이얼로그 | 컨텍스트 전환 없이 작업할 때 |
| 내비게이션 | 사이드바, 브레드크럼, 탭 설계 시 |
| 검색/필터 | 데이터 탐색 기능이 있을 때 |
| 드래그 앤 드롭 | 순서 변경, 칸반, 파일 업로드 시 |
| Undo/Redo | 되돌리기가 필요한 편집 기능 시 |
| 권한 기반 UI | 역할별 접근 제어가 있을 때 |

### Step 3. 실무 수치 확인 (세부 구현 시)

`web-app-practical.md`를 읽고 구체적인 수치와 접근법을 적용하라.

| 주제 | 언제 참조하는가 |
|-----|---------------|
| 접근성 | 항상. 포커스 관리, 키보드 내비, ARIA, 색상 대비 |
| 간격 체계 (4가지 접근법) | 레이아웃 간격을 결정할 때 |
| 애니메이션 | 전환/인터랙션 타이밍을 정할 때 |
| 반응형 브레이크포인트 | 다중 디바이스 대응 시 |
| 대시보드 패턴 | KPI 카드, 차트, 위젯 배치 시 |

### Step 4. 마감 검증 (필수, 최종)

`visual-and-verification.md`를 읽고 미학적 방향과 검증 체크리스트를 적용하라. 플랫폼과 무관하게 모든 디자인의 마감 단계에서 반드시 거쳐야 한다.

## 핵심 판단 체크리스트

설계 중 막힐 때 이 질문으로 돌아오라:

- [ ] 이 화면의 지배적 패러다임은 무엇인가?
- [ ] 사용자의 주된 행동 하나는 무엇인가?
- [ ] 사이드바가 필요한가, 아니면 전체 화면이 나은가?
- [ ] 정보 밀도는 의도적으로 선택했는가? (Compact/Medium/Airy)
- [ ] 키보드만으로 핵심 워크플로우를 완수할 수 있는가?
