# 모바일 애플리케이션 디자인 — 플랫폼 개요

> 이 문서는 모바일 앱 디자인 작업 시 **반드시 먼저 읽는 진입점**이다. 세부 가이드로 바로 가지 말고, 여기서 방향을 잡은 뒤 필요한 문서를 순서대로 참조하라.

---

## 모바일 앱이란

손 안의 디바이스에서 동작하는 앱. 터치 기반, 제한된 화면, 이동 중 사용이 기본 전제. 30초 마이크로 태스크부터 30분 몰입 세션까지 사용 패턴이 극단적으로 다르다.

---

## 설계 워크플로우 — 이 순서를 따르라

### Step 1. 패러다임 선택 (필수, 최우선)

`mobile-app-paradigms.md`를 읽고 **앱의 지배적 패러다임을 하나 선택하라.**

| 패러다임 | 핵심 전제 | 대표 사례 |
|---------|----------|----------|
| **구조-수직 스택** | 탭 바 + 독립 내비 스택. 화면은 계층적 | 설정, 은행, 이메일 |
| **피드-몰입** | 전체 화면 콘텐츠. UI 최소화. 스와이프 진행 | TikTok, Instagram, Spotify |
| **레이어-맵 컨텍스트** | 고정 배경 위 바텀시트 레이어 (peek/half/full) | Apple Maps, Uber |
| **대화형** | 채팅 인터페이스가 주 인터랙션. 리치 카드 출력 | ChatGPT, WeChat |

> **판단 기준**: "사용자가 이 앱에서 시간을 어떻게 보내는가?"
> - 명확한 기능 탐색과 CRUD → 구조-수직 스택
> - 콘텐츠 소비에 몰입 → 피드-몰입
> - 지도/공간 위에서 컨텍스트 탐색 → 레이어-맵 컨텍스트
> - 질문-응답, 메시지 기반 상호작용 → 대화형

하나의 앱 안에서도 화면마다 패러다임이 다를 수 있다. **하지만 한 화면에 두 패러다임을 섞지 말라.**

### Step 2. 컴포넌트 패턴 적용 (구현 시)

`mobile-app-components.md`를 읽고 화면에 필요한 컴포넌트의 설계 규칙을 따르라.

| 컴포넌트 | 언제 참조하는가 |
|---------|---------------|
| 제스처 | 스와이프, 핀치, 풀투리프레시 등 터치 인터랙션 설계 시 |
| 엄지 영역 | 버튼/CTA 배치 위치를 결정할 때 |
| 폼 | 사용자 입력을 받는 모든 화면 |
| 온보딩 | 첫 실행 경험을 설계할 때 |
| 리스트 | 목록 형태 데이터 표시, 스와이프 액션 시 |
| 바텀시트 | 컨텍스트 유지하며 추가 정보/행동 노출 시 |
| 액션시트/컨텍스트 메뉴 | 선택지 목록을 제시할 때 |
| 알림/피드백 | 토스트, 배너, 햅틱 피드백 설계 시 |
| 검색 | 검색 기능이 있는 화면 |
| 오프라인/연결 | 네트워크 불안정 상황 대응 시 |
| 로딩 상태 | 스켈레톤, 스플래시, 부분 로딩 시 |
| 이미지/미디어 | 이미지 뷰어, 그리드, 비디오 플레이어 시 |

### Step 3. 실무 수치 확인 (세부 구현 시)

`mobile-app-practical.md`를 읽고 구체적인 수치와 플랫폼 차이를 적용하라.

| 주제 | 언제 참조하는가 |
|-----|---------------|
| 접근성 | 항상. Dynamic Type, VoiceOver/TalkBack, 터치 타겟, 색상 대비 |
| iOS vs Android 차이 | 플랫폼별 구현 시. 내비게이션, 상태바, 타이포, 모달, 아이콘 |
| 간격 체계 (3가지 접근법) | 레이아웃 간격을 결정할 때 |
| 내비게이션 패턴 | 탭 바, 드로어, 제스처 내비, 바텀시트 설계 시 |
| 애니메이션/전환 | push/pop/modal 전환 타이밍 정할 때 |
| 성능 | 로딩 목표치, 이미지 최적화, 앱 시작 시간, 메모리 관리 |

### Step 4. 마감 검증 (필수, 최종)

`visual-and-verification.md`를 읽고 미학적 방향과 검증 체크리스트를 적용하라. 플랫폼과 무관하게 모든 디자인의 마감 단계에서 반드시 거쳐야 한다.
