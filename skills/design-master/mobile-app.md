# 모바일 애플리케이션 디자인 가이드

> 모바일 설계의 근본 질문: "사용자가 이 앱에서 시간을 어떻게 보내는가" 30초 마이크로 태스크인가, 30분 몰입 세션인가에 따라 패러다임이 달라진다.

## 패러다임 A: 구조-수직 스택 — 목적지 기반 내비게이션

전통적 모바일 앱. "앱은 여러 목적지(탭)의 집합이고, 각 탭은 독립적인 수직 스택." 설정, 은행, 이메일 등 **명확한 기능 구분** 앱.

핵심 사상: **화면은 계층적. 위로 갈수록 넓고(목록), 아래로 갈수록 좁다(상세).**

```
┌─────────────────────────┐
│   1) 상태 바 (OS 영역)   │  62px, 터치 불가
├─────────────────────────┤
│                         │
│   2) 앱 콘텐츠           │  유일한 작업 영역
│                         │
├─────────────────────────┤
│   3) 하단 바 (선택)      │  탭 바 / 하단 내비게이션
└─────────────────────────┘

내비게이션 흐름:
  탭 A → 목록 → 상세 → 편집 (push/pop 스택)
  각 탭은 독립적 스택 — 탭 전환 시 이전 스택 보존
```

설계 규칙:
- **화면당 하나의 주된 의도**
- **한 손 사용 설계**: 주요 행동은 하단 절반
- 터치 타겟: 최소 44×44pt
- 단일 수직 스크롤: 중첩 스크롤 회피
- 하단 탭 바: **3~5개 탭** — 최상위 목적지만

**적합**: 유틸리티, 금융, 이커머스, 설정 중심 앱.

## 패러다임 B: 몰입-수직 피드 — 콘텐츠가 곧 인터페이스

**콘텐츠 자체가 전체 화면을 점유**, 스와이프/스크롤로 이동. "앱은 콘텐츠의 무한 스트림."

핵심 사상: **UI를 최소화하라. 콘텐츠가 100%, 인터페이스는 유령.**

```
참조: TikTok, Instagram Reels, YouTube Shorts, Spotify

┌─────────────────────────┐
│                         │
│    콘텐츠 (100% 화면)    │
│                         │
│              ┌─────┐    │
│              │ 좋아요│    │  우측 하단: 플로팅 액션
│              │ 댓글 │    │  반투명, 콘텐츠 위 겹침
│              │ 공유 │    │
│              └─────┘    │
│  @이름 · 설명...         │  하단: 오버레이 메타데이터
└─────────────────────────┘

인터랙션 문법:
  위로 스와이프    = 다음 콘텐츠
  아래로 스와이프  = 이전 콘텐츠
  탭             = 재생/정지
  더블 탭         = 좋아요
  길게 누르기     = 컨텍스트 메뉴
```

설계 규칙:
- **크롬(chrome)을 없애라**: edge-to-edge
- 메타데이터는 오버레이: 반투명 배경/텍스트 그림자
- 전환이 곧 내비게이션: 스와이프 제스처 = 다음 페이지
- 자동 재생 + 자동 진행

**적합**: 소셜 미디어, 숏폼 비디오, 미디어 소비, 데이팅 앱.

## 패러다임 C: 맥락-레이어 — 바텀 시트와 오버레이 중심

고정된 배경 위에 **바텀 시트, 모달이 겹겹이**. "메인 컨텍스트를 유지하면서 부가 정보 접근."

핵심 사상: **배경이 앵커, 디테일은 레이어로. 화면 전환 대신 레이어 전환.**

```
참조: Apple Maps, Uber, Google Maps, 배달의민족

┌─────────────────────────┐
│                         │
│   배경 컨텍스트           │  지도, 목록 등
│   (항상 보인다)           │
│                         │
├──── ─── ─── ────┤  ← drag handle (36×4px)
│                         │
│   바텀 시트               │  3단계: peek / half / full
│                         │
└─────────────────────────┘

바텀 시트 3단계:
  Peek  = 화면의 15-25% (검색 바, 요약만)
  Half  = 화면의 50% (목록, 상세)
  Full  = 화면의 90% (상단 64px 남겨 배경 힌트)

레이어 우선순위 (아래→위):
  1. 배경 컨텍스트
  2. 바텀 시트
  3. 액션 시트
  4. 모달 다이얼로그
  5. 토스트/스낵바
```

설계 규칙:
- **배경은 절대 완전히 사라지지 않는다**
- 시트 드래그 vs 내부 스크롤 구분
- 각 레이어는 독립적으로 해제 가능
- dimming 오버레이 rgba(0,0,0,0.4)
- 시트 내부에서 추가 시트 금지 (최대 2단계)

**적합**: 지도/위치 기반, 차량 호출, 배달, 부동산, 여행 예약.

## 패러다임 D: 대화-주도(Conversational) — 채팅이 곧 인터페이스

**대화(chat) 인터페이스가 주 상호작용**. AI 에이전트, 챗봇, 메시징 앱.

핵심 사상: **사용자는 말로 의도를 전달. 시스템이 구조화된 결과를 돌려준다.**

```
참조: ChatGPT, Claude, 카카오톡 미니앱, WeChat

┌─────────────────────────┐
│  [←] 대화 제목    [···]  │
├─────────────────────────┤
│  시스템: [날씨 카드 UI]   │  ← 리치 카드
│  ┌─────────────────┐    │
│  │ 서울 · 맑음 22°  │    │
│  │ [상세 보기]      │    │
│  └─────────────────┘    │
├─────────────────────────┤
│  [+] 메시지 입력...  [→] │
└─────────────────────────┘

리치 응답 유형:
  - 텍스트 버블, 카드, 캐러셀, 퀵 리플라이 칩
  - 폼, 코드 블록, 테이블
```

설계 규칙:
- **입력은 단순, 출력은 풍부**
- 퀵 리플라이 칩으로 다음 행동 제안
- 긴 응답은 스트리밍(타이핑 중...) 표시
- **탈출구 제공**: 대화 불가 시 전통적 UI로 전환

**적합**: AI 어시스턴트, 고객 지원, 메시징, 음성 인터페이스.

---

> **모바일 패러다임도 배타적이지 않다.** Uber = 맥락-레이어 + 대화. Instagram = 몰입-피드 + 구조-스택 + 대화. **각 화면의 지배적 패러다임을 의식하는 것**이 핵심.

## 공통 모바일 규칙

```
DO:
✓ 핵심 CTA를 스크롤 없이 볼 수 있게
✓ 복잡한 그리드보다 단순 스택 선호
✓ 래퍼의 gap으로 모든 수직 간격 관리
✓ 하단 여백은 padding-bottom으로

DON'T:
✗ 폴드 위에 경쟁하는 여러 섹션 배치
✗ 섹션별 개별 수평 패딩 추가 (래퍼가 처리)
✗ 하단 빈 공간에 스페이서 요소 사용
✗ 핵심 행동을 접근 어려운 구석에 숨기기
```

## 컴포넌트별 상세 설계 패턴

모바일은 데스크톱과 다르다. 화면이 작고, 입력이 부정확하며, 맥락이 끊긴다. "데스크톱 UI를 줄이면 모바일"이 아니다. **모바일은 처음부터 다른 사고방식으로 설계해야 한다.**

### 제스처 문법

제스처는 모바일의 핵심 언어다. 그러나 보이지 않는 인터랙션이므로 **발견 가능성(discoverability)**이 항상 도전이다.

```
기본 제스처 (모든 사용자가 알고 있다 — 설명 불필요):
  탭(Tap)        = 선택, 활성화
  스크롤(Scroll)  = 콘텐츠 탐색
  핀치(Pinch)     = 확대/축소

학습된 제스처 (대부분 알지만 힌트 필요):
  스와이프(Swipe)      = 삭제, 아카이브, 다음 항목
  풀 투 리프레시       = 데이터 새로고침
  길게 누르기(Long Press) = 컨텍스트 메뉴, 재정렬 모드

고급 제스처 (반드시 대안 UI 필요):
  엣지 스와이프(Edge Swipe) = 뒤로 가기, 드로어 열기
  더블 탭               = 좋아요, 확대 토글
  3D Touch / Haptic Touch = 미리보기, 퀵 액션
  핀치 아웃(Spread)      = 사진 확대, 지도 줌인

스와이프 행동 설계:
  ┌────────────────────────────────────┐
  │  ← [삭제 (빨강)]  항목  [아카이브 (파랑)] →  │
  └────────────────────────────────────┘

  좌측 스와이프: 파괴적 행동 (삭제, 신고)
  우측 스와이프: 비파괴적 행동 (아카이브, 읽음 표시)

  스와이프 임계값:
    살짝 스와이프 (30-40%) = 행동 버튼 노출
    풀 스와이프 (70%+) = 즉시 실행
    풀 스와이프 파괴적 행동: 확인 단계 추가하거나 Undo 토스트

  스와이프 피드백:
    행동 아이콘 + 배경색 + 햅틱(가능 시)
    탄성(rubber-band): 끝까지 스와이프 후 놓으면 원위치로 bounce

풀 투 리프레시:
  트리거: 스크롤 최상단에서 60-80px 이상 당기기
  피드백: 스피너 등장 (당기는 동안 회전 시작)
  완료: 스피너 사라짐 + 콘텐츠 업데이트
  안티패턴: 타이머 기반 자동 새로고침과 혼합하지 말라
  안티패턴: 중첩 스크롤 뷰에서 풀 투 리프레시는 오동작한다

  규칙: 풀 투 리프레시는 항상 물리적 은유를 따라야 한다
  규칙: 새로고침 중 사용자 조작을 차단하지 말라
```

### 엄지 영역(Thumb Zone) 설계

```
┌────────────────────┐
│                    │  어려움: 한 손으로 닿기 힘든 영역
│   ████████████     │  → 드문 행동, 내비게이션 바
│                    │
│   ░░░░░░░░░░░░     │  보통: 약간 노력 필요
│                    │  → 보조 행동, 검색
│   ▓▓▓▓▓▓▓▓▓▓▓▓     │
│                    │  자연스러움: 쉽게 닿는 영역
│   ▓▓▓▓▓▓▓▓▓▓▓▓     │  → 주 행동, 하단 탭, FAB
└────────────────────┘

배치 원칙:
  주 행동(Primary CTA): 화면 하단 1/3
  보조 행동: 화면 중앙
  드문 행동(설정, 프로필): 화면 상단
  파괴적 행동: 쉬운 접근 구역에 두지 말라 (실수 방지)

큰 화면(6.5"+) 추가 고려:
  좌측 상단이 가장 닿기 어렵다 → 햄버거 메뉴를 좌상단에 두는 건 나쁜 패턴
  해결: 하단 시트 기반 내비게이션, 우하단 FAB, 엣지 제스처
```

### 모바일 폼 설계

모바일에서 폼은 사용자가 가장 이탈하기 쉬운 구간이다. **입력을 최소화하고, 키보드를 현명하게 관리하라.**

```
키보드 관리:
  키보드 유형을 입력에 맞춰라:
    이메일    → type="email" (@ 키 표시)
    전화번호  → type="tel" (숫자 패드)
    URL      → type="url" (. / .com 키)
    숫자     → inputmode="numeric" (순수 숫자 패드)
    검색     → type="search" (리턴 키가 "검색"으로)

  키보드 높이 대응:
    키보드가 올라오면 포커스된 필드가 보이도록 스크롤 조정 필수
    하단 고정 버튼: 키보드 위로 올라오거나, 스크롤 내 배치로 전환
    안전 영역: iOS에서 키보드 + 제안 바 높이까지 고려

입력 최소화 전략:
  자동 완성: autocomplete 속성 적극 활용
    name, email, tel, street-address, postal-code, cc-number...

  선택형 > 입력형:
    날짜 → 날짜 피커 (네이티브 또는 커스텀)
    성별 → 세그먼트 컨트롤 (2-4개 선택지)
    국가 → 검색 가능한 드롭다운
    수량 → 스테퍼 (- / + 버튼)
    범위 → 슬라이더

  스마트 기본값:
    위치 기반 국가/시간대 자동 선택
    최근 입력값 제안
    카메라로 신용카드/주소 스캔

필드 레이아웃:
  단일 열 필수: 모바일에서 2열 폼은 금지
  필드 높이: 48-56px (터치 타겟 충족)
  필드 간격: 16-20px
  레이블: 필드 상단, 12-14px (floating label 허용)
  에러: 필드 바로 아래, destructive 색, 12-13px

  Floating Label 패턴:
    빈 필드: 레이블이 필드 내부에 (placeholder 역할)
    포커스/입력 시: 레이블이 상단으로 올라감 (12px, primary 색)
    장점: 공간 절약
    단점: placeholder와 레이블 혼동 가능 — 접근성 주의

CTA 버튼:
  위치: 폼 하단, 전체 너비
  높이: 52-56px
  키보드 열림 시: 버튼이 키보드에 가려지면 안 됨
    → 스크롤 가능하게 하거나, 키보드 위에 고정

패스워드 필드:
  "보기/숨기기" 토글 아이콘 필수
  비밀번호 강도 표시기: 실시간 색상 바 (빨강→노랑→초록)
  요구사항 체크리스트: 입력하면서 하나씩 ✓로 전환
```

### 온보딩 패턴

```
유형별 패턴:

A. 워크스루 캐러셀 (3-5장):
  ┌──────────────────────┐
  │                      │
  │     [일러스트]        │  60% 높이
  │                      │
  │  가치 제안 헤드라인    │  20-24px Bold
  │  간단한 설명 한 줄     │  14-16px Regular
  │                      │
  │  ● ○ ○               │  페이지 인디케이터
  │  [건너뛰기]  [다음 →]  │
  └──────────────────────┘

  규칙:
    3장이 이상적, 5장 이하 필수 (3장 넘어가면 완주율 급락)
    "건너뛰기" 항상 제공 (강제 온보딩 금지)
    마지막 장: CTA 버튼 ("시작하기")
    안티패턴: 기능 나열. "이 기능이 있습니다" → "이 문제가 해결됩니다"

B. 코치마크(Coach Mark):
  특정 UI 위에 스포트라이트 + 말풍선으로 설명
  규칙:
    한 번에 하나만 표시
    탭으로 다음 단계 진행
    총 3-5단계 이하
    첫 실행 시에만 표시 (반복 금지)
    스킵 항상 가능
    이미 사용 중인 사용자에게는 불필요한 코치마크 노출하지 말라

C. 프로그레시브 온보딩:
  첫 실행 시 모든 것을 설명하지 않는다
  기능에 처음 접근할 때마다 해당 기능만 설명
  적합: 복잡한 앱 (Figma, Notion, Slack)

  구현:
    사용자 행동 추적 → 미경험 기능 식별
    해당 기능 첫 접근 시 → 인라인 팁 또는 빈 상태에 가이드 포함
    경험 후 → 팁 영구 숨김

D. 인터랙티브 튜토리얼:
  실제 앱 UI에서 직접 행동하게 한다 (읽는 것이 아니라 하는 것)
  적합: 제스처 기반 앱, 게임, 복잡한 도구

  구현:
    단계별 작업 제시 → 완료하면 다음 단계 활성화
    잘못된 조작 시 부드러운 안내 (좌절 아닌 학습)
    진행 상황 저장 (중단 후 재개 가능)

퍼미션 요청:
  절대 앱 실행 즉시 모든 권한을 동시에 요청하지 말라
  올바른 흐름:
    1. 기능 사용 시점까지 지연 (Just-in-Time)
    2. 시스템 다이얼로그 전에 이유 설명 (Pre-permission screen)
    3. 거부 시: 기능 제한 상태로 진행 + 나중에 설정에서 변경 안내

  예시:
    카메라: "프로필 사진을 찍으려면 카메라 접근이 필요합니다"
    → [나중에] [허용하기] → (시스템 다이얼로그)
    알림: 첫 의미 있는 이벤트 발생 후 "알림을 받으시겠습니까?"
```

### 모바일 리스트 패턴

```
기본 리스트:
  항목 높이: 48-72px (내용에 따라)
  좌측: 아바타/아이콘 (40-48px) + 12px 간격
  중앙: 제목 (16px/500) + 부제목 (14px/400, muted)
  우측: 메타데이터(시간, 배지) + 셰브론(>)
  구분선: 1px, 좌측 패딩 만큼 들여쓰기 (리딩 콘텐츠에서 시작)
  그룹 헤더: 32-40px, 12-14px uppercase, 배경 surface-1

섹션 리스트 (iOS 설정 스타일):
  인셋 그룹: 좌우 16px 마진, 상하 8-12px 마진
  radius: 10-12px (그룹 전체)
  배경: 카드(surface-1) / 페이지 배경(surface-0)

스와이프 액션 리스트:
  좌측 스와이프: 1-2개 행동 (삭제, 아카이브)
  우측 스와이프: 1-2개 행동 (읽음 표시, 핀)
  전체 스와이프: 하나의 빠른 행동

리오더(Reorder) 리스트:
  핸들: 우측 ≡ 아이콘 (길게 누르기 또는 편집 모드)
  드래그 중: 항목 약간 확대(scale 1.02) + 그림자 + 배경색 변화
  드롭: 200ms ease-out 안착 애니메이션
  다른 항목: 150ms로 위아래 밀려남

무한 스크롤 리스트:
  하단 로딩: 스피너(24px) + "로딩 중..." 텍스트
  트리거: 하단에서 3-5개 항목 남았을 때 프리페치
  에러: "로딩 실패" + [다시 시도] 버튼 (인라인)
  끝: "모든 항목을 확인했습니다" 메시지

  성능:
    셀 재사용 필수 (RecyclerView / UITableView 패턴)
    이미지: 스크롤 중 저해상도 → 정지 후 고해상도 로드
    문자열 truncation: 2줄 이하 + ellipsis
```

### 바텀 시트 상세 설계

```
구조:
  ┌────────────────────────┐
  │      ─────── (핸들)     │  핸들: 36×4px, radius 2px, 중앙
  │                        │  핸들 영역: 높이 20px, 드래그 인식 영역
  ├────────────────────────┤
  │  제목            [닫기]  │  헤더: 48-56px, 선택적
  ├────────────────────────┤
  │                        │
  │  콘텐츠 영역             │  스크롤 가능
  │                        │
  └────────────────────────┘

스냅 포인트:
  Peek  = 화면의 15-25% (검색 바, 요약만)
  Half  = 화면의 50% (목록, 상세)
  Full  = 화면의 90% (상단 64px 남겨 배경 힌트)

  규칙: 최소 2개 스냅 포인트 (보통 3개)
  규칙: 중간에서 놓으면 가장 가까운 스냅 포인트로 이동 (velocity 고려)
  규칙: 빠르게 아래로 스와이프하면 Peek나 닫힘으로 직행 (velocity threshold)

드래그 vs 스크롤 충돌 해결:
  콘텐츠 최상단에서 아래로 드래그 → 시트 축소
  콘텐츠 중간에서 아래로 드래그 → 콘텐츠 스크롤

  구현: 콘텐츠 scrollTop === 0 일 때만 시트 드래그 활성화
  iOS: UISheetPresentationController가 자동 처리
  Android: BottomSheetBehavior의 nestedScrolling 설정

배경 인터랙션:
  Peek 상태: 배경 탭/조작 가능 (지도 패닝 등)
  Half 상태: 배경 탭 가능, 딤 없음
  Full 상태: 배경 탭 시 시트 닫힘 또는 Half로 축소

  안티패턴: 시트 안에서 또 다른 시트를 여는 것 (최대 2단계)
  대안: Full 시트 내부에서 내비게이션 push/pop 사용
```

```
    ┌────────────────────────┐

    │  🖼 앨범에서 선택         │  높이: 56px

  Android 스타일:

  공통 규칙:
    항목 수: 3-8개 (초과 시 구조 재설계)
    아이콘 + 텍스트 (아이콘만 사용 금지)
```

### 모바일 알림 & 피드백

```
토스트/스낵바:
  위치: 하단, 탭 바 위 + safe area 위
  너비: 전체 너비 - 좌우 16px 마진
  높이: 48-56px

  안티패턴: 에러를 토스트로 표시 (에러는 인라인 또는 전체 화면 상태)

인앱 배너:
  위치: 화면 상단, 상태 바 아래
  높이: 자동 (콘텐츠에 따라), 최소 48px
  아래로 슬라이드 인 / 위로 슬라이드 아웃
  스와이프로 닫기 허용

햅틱 피드백:
  경량(Light): 토글 전환, 선택 변경
  중간(Medium): 행동 완료, 스냅 포인트 도달
  강한(Heavy): 파괴적 행동 확인, 에러
  성공(Success): 완료, 체크아웃
  경고(Warning): 주의 필요
  에러(Error): 실패, 잘못된 입력

  규칙: 과도한 햅틱은 배터리 소모 + 사용자 피로
  규칙: 햅틱 = 시각적 피드백의 보조, 단독으로 의미 전달 금지
  접근성: 햅틱을 끈 사용자도 동일하게 사용 가능해야 한다
```

### 오프라인 & 연결 상태

```
연결 상태 표시:
  오프라인 전환: 화면 상단 인앱 배너 "인터넷 연결이 없습니다"
    배경: warning 색 (노랑) 또는 회색
    닫기 불가 (연결 복원까지 유지)
    하단 탭/내비게이션은 계속 작동

  온라인 복원: 배너 "연결되었습니다" (초록) → 3초 후 자동 사라짐
    자동 동기화 시작 (사용자 알림 없이)

오프라인 모드 설계:
  레벨 1 — 읽기 전용: 캐시된 데이터 표시, 쓰기 차단
    행동 버튼: disabled + 툴팁 "오프라인에서는 사용할 수 없습니다"

  레벨 2 — 큐 기반: 오프라인 중 행동을 큐에 저장
    시각 표시: ☁️ 아이콘 → 동기화 대기 중
    복원 시: 순서대로 실행 + 충돌 해결 UI

  레벨 3 — 완전 오프라인: 로컬 DB (SQLite) + 동기화 엔진
    적합: 노트앱, 할일앱, 메시징 (드래프트)
    충돌: 마지막 쓰기 우선(Last Write Wins) 또는 사용자 선택

캐시 전략:
  API 응답: stale-while-revalidate (캐시 표시 → 백그라운드 갱신)
  이미지: 디스크 캐시 + 메모리 캐시 2단계
  지도 타일: 미리 다운로드 허용 (오프라인 지도)
```

```
스켈레톤 스크린:

프로그래스:
  업로드/다운로드: 진행 바 + 퍼센트 + 남은 시간
  처리 중: indeterminate 프로그래스 바 + "처리 중..." 텍스트
  위치: 콘텐츠 인라인 또는 하단 시트
```

## 접근성 상세 가이드

"모바일 접근성은 데스크톱보다 더 중요하다. 모바일은 모든 상황에서 사용된다 — 밝은 햇빛, 한 손, 이동 중, 소음 속."

```
Dynamic Type (iOS) / 텍스트 크기 조절 (Android):
  모든 텍스트는 시스템 크기 설정을 존중해야 한다
  테스트: 가장 큰 크기로 설정하고 레이아웃 깨짐 확인
  규칙:
    고정 높이 컨테이너에 텍스트 넣지 말라 (잘림 위험)
    줄 수 제한 시: numberOfLines + lineBreakMode (truncation)
    최소 텍스트 크기: 11pt (이보다 작으면 읽기 어려움)

VoiceOver (iOS) / TalkBack (Android):
  모든 인터랙티브 요소: accessibilityLabel 필수
  이미지:
    의미 있는 이미지: 내용 설명 (예: "팀 미팅 중인 사람들")
    장식 이미지: accessibilityElementsHidden = true
  커스텀 컴포넌트:
    accessibilityRole 명시 (button, link, header, image...)
    상태 변화: accessibilityValue 업데이트

  테스트: 눈 감고 VoiceOver/TalkBack만으로 주요 플로우 완수할 수 있는가?

터치 타겟:
  최소: 44×44pt (iOS) / 48×48dp (Android)
  인접 타겟 간격: 최소 8px (실수 방지)
  작은 시각적 요소도: 터치 영역은 최소 크기 유지 (padding으로)

  안티패턴: 시각적 크기 = 터치 영역 (아이콘 16px → 터치 영역도 16px)
  올바름: 시각적 16px 아이콘 + 투명 padding으로 48px 터치 영역

색상 & 대비:
  텍스트: 4.5:1 이상 (WCAG AA)
  대형 텍스트 (18px+): 3:1 이상
  UI 컴포넌트: 3:1 이상

  야외 사용 고려:
    햇빛 아래에서도 읽을 수 있는가?
    대비 높은 모드 테스트 필수

모션:
  Reduce Motion 설정 존중 (iOS: UIAccessibility.isReduceMotionEnabled)
  대안: 화려한 전환 → 심플한 페이드 또는 즉시 전환
  Parallax 효과: Reduce Motion에서 완전 비활성화
  자동 재생 미디어: 자동 정지
```

## iOS vs Android 플랫폼 차이

두 플랫폼은 철학이 다르다. "하나의 디자인으로 양쪽을 커버하겠다"는 나이브한 접근이다. 최소한 아래 차이를 인지하라.

```
내비게이션:
  iOS: 좌측 엣지 스와이프 → 뒤로 가기 (시스템 제스처)
  Android: 하단 백 버튼/제스처 → 뒤로 가기

  iOS: 내비게이션 바 좌측 [< 뒤로] 텍스트
  Android: 내비게이션 바 좌측 ← 화살표 아이콘

  결론: iOS 스와이프 뒤로를 차단하면 사용자가 혼란

상태 바:
  iOS: Dynamic Island (62px) / 노치 (47px) / 일반 (20px)
  Android: 24dp 고정
  → safe area 상단 처리 필수 (env(safe-area-inset-top))

하단 영역:
  iOS: 홈 인디케이터 34pt (safe area bottom)
  Android: 제스처 내비게이션 바 또는 3버튼 바
  → 하단 고정 요소는 safe area 위에 배치

타이포그래피:
  iOS: SF Pro (시스템 기본), Dynamic Type 시스템
  Android: Roboto (시스템 기본), Material Type Scale
  크로스 플랫폼: 시스템 폰트 사용 권장 또는 커스텀 폰트 양쪽 동일 적용

모달/시트:
  iOS: pageSheet (.formSheet) — 상단 둥근 모서리, 카드형
  Android: BottomSheet — 하단에서 올라옴, 풀스크린 가능

  iOS 16+: UISheetPresentationController (스냅 포인트 자동)
  Android: Material BottomSheetDialogFragment

날짜/시간 피커:
  iOS: 휠 스타일 또는 인라인 캘린더
  Android: Material DatePicker (캘린더 뷰)
  → 네이티브 피커 사용 권장 (사용자 익숙함)
  크로스 플랫폼: 커스텀 피커 시 양쪽 관례를 모두 테스트

알림:
  iOS: 배너 → 알림 센터 (스와이프로 행동)
  Android: 알림 패널 (인라인 행동 버튼, 진행 바 가능)

  iOS 10+: Rich Notification (이미지, 버튼)
  Android: Notification Channels (카테고리별 제어)

앱 아이콘:
  iOS: 1024×1024px, 라운드 사각형(자동 마스킹), 그림자 없음
  Android: 1024×1024px, Adaptive Icon (전경 + 배경 레이어)
  → 아이콘의 핵심 요소를 중앙 66% 안에 배치 (마스킹 대비)
```

## 실무 수치

### 접근법 A: iOS HIG 기반

```
고정 치수:
  상태 바         = 54px (Dynamic Island), 47px (노치), 20px (구형)
  내비게이션 바   = 44pt (Large Title +52pt = 96pt)
  탭 바           = 49pt + 하단 safe area 34pt
  콘텐츠 좌우 마진 = 16pt (컴팩트), 20pt (레귤러)

터치 타겟:
  최소 = 44×44pt / 권장 = 48×48pt / 버튼 간 최소 간격 = 8pt

타이포 (SF Pro, Dynamic Type):
  Large Title = 34pt Bold    Headline    = 17pt Semibold
  Title 1     = 28pt Bold    Body        = 17pt Regular
  Title 2     = 22pt Bold    Subheadline = 15pt Regular
  Title 3     = 20pt Semibold Footnote   = 13pt Regular
  Caption 1   = 12pt Regular Caption 2   = 11pt Regular

색상:
  systemBlue  = #007AFF (라이트) / #0A84FF (다크)
  systemRed   = #FF3B30 / #FF453A
  systemGreen = #34C759 / #30D158
  separator   = rgba(60,60,67,0.29) / rgba(84,84,88,0.6)
  배경 grouped = #F2F2F7 / #000000
```

### 접근법 B: Material Design 3 기반

```
고정 치수:
  상태 바      = 24dp
  Top App Bar  = 64dp(Small), 152dp(Medium), 280dp(Large)
  Bottom Nav   = 80dp
  FAB          = 56×56dp / 확장 96×96dp

터치 타겟: 최소 48×48dp

타이포 (Material 3):
  Display Large  = 57sp / line-height 64sp / tracking -0.25
  Display Medium = 45sp / 52sp
  Display Small  = 36sp / 44sp
  Headline Large = 32sp / 40sp
  Title Large    = 22sp / 28sp
  Body Large     = 16sp / 24sp / tracking +0.50
  Body Medium    = 14sp / 20sp / tracking +0.25
  Label Large    = 14sp / 20sp / tracking +0.10

Elevation: 0dp(배경) → 1dp(카드) → 3dp(버튼) → 6dp(FAB) → 8dp(nav) → 12dp(다이얼로그)

Shape: None(0) → ExtraSmall(4dp) → Small(8dp) → Medium(12dp) → Large(16dp) → ExtraLarge(28dp) → Full(50%)
```

### 접근법 C: 커스텀 디자인 시스템

```
간격 (6px 베이스):
  xxs = 2px   sm  = 12px   xl  = 36px
  xs  = 4px   md  = 18px   2xl = 48px
  s   = 6px   lg  = 24px   3xl = 72px

타이포 (브랜드 폰트):
  Micro    = 10px/600/uppercase/tracking +1.5px
  Caption  = 12px/400/tracking +0.3px
  Body     = 16px/400/line-height 1.5
  Title    = 24px/700/tracking -0.5px
  Hero     = 32px/800/tracking -1.0px

컴포넌트:
  리스트 아이템 = 56px 높이
  카드          = padding 16px, radius 12px
  인풋 필드     = 48px 높이, radius 8px
  Bottom Sheet  = radius 상단 20px, drag handle 36×4px
```

## 모바일 내비게이션 패턴 비교

```
패턴 A — 하단 탭 바 (Instagram, Uber):
  3-5개 최상위 목적지, 항상 표시, 한 손 접근 용이
  49pt(iOS) / 80dp(Android)

  규칙: 현재 탭 재탭 = 루트로 스크롤 + 스택 초기화
  규칙: 탭 아이콘 아래 레이블 필수 (아이콘만은 접근성 위반)
  규칙: 각 탭 아이콘은 활성/비활성 상태 구분 (filled vs outlined)

패턴 B — 햄버거 + 드로어 (Gmail, Slack):
  목적지 5개 초과 시, 계층적 내비게이션

  규칙: 드로어 너비 = 화면의 85% (최대 360px), 나머지는 딤 오버레이
  규칙: 엣지 스와이프 또는 햄버거 아이콘으로 열기
  안티패턴: 핵심 기능을 드로어 안에 숨기기 (발견하기 어렵다)

패턴 C — 제스처 기반 (Tinder, 지도):
  몰입형 경험, 카드 탐색, 미디어 소비

  규칙: 제스처 힌트(onboarding)가 반드시 필요
  규칙: 제스처의 대안 UI 항상 제공 (버튼, 메뉴)

패턴 D — 바텀 시트 기반 (Apple Maps, Uber):
  맥락 보존 + 상세 정보 동시 노출
  최대: 화면 - 64px 상단 여유

  규칙: 드래그 핸들이 보여야 시트가 조작 가능하다는 것을 인지
  규칙: 키보드가 올라오면 시트도 함께 올라가야 한다
```

## 모바일 애니메이션 & 트랜지션

```
화면 전환:
  Push (일반): 우→좌 슬라이드 300ms ease-out
  Pop (뒤로): 좌→우 슬라이드 250ms ease-in-out
  Modal 올림: 아래→위 슬라이드 350ms spring
  Modal 닫기: 위→아래 슬라이드 250ms ease-in
  탭 전환: 크로스 페이드 150ms (슬라이드 아님)

  iOS: UINavigationController 기본 전환은 interactive pop gesture와 연결
  Android: Material Shared Axis 또는 Fade Through

공유 요소 전환:
  목록 → 상세: 선택한 이미지/카드가 확대되며 상세 화면으로
  iOS: UIViewControllerAnimatedTransitioning
  Android: Shared Element Transition

  규칙: 전환 시작점과 끝점을 명확히 연결하라 (공간적 연속성)
  규칙: 전환 중 다른 요소는 페이드 아웃 → 전환 후 페이드 인

마이크로인터랙션:
  토글 전환: 200ms spring
  체크박스 체크: 150ms + 체크마크 그리기 애니메이션
  좋아요: 300ms scale(1→1.3→1) + 색상 전환 + 파티클(선택)
  삭제 스와이프: 항목 사라짐 300ms + 아래 항목 올라옴 200ms
  풀 투 리프레시 스피너: 당김에 비례한 회전 → 릴리즈 후 자동 회전

  60fps 유지 규칙:
    애니메이션은 transform + opacity만 사용 (레이아웃 변경 최소화)
    긴 리스트 스크롤 중 복잡한 애니메이션 피하라
    그림자 변경은 비용이 높다 → 가능하면 사전 렌더링
```

## 모바일 성능 & 체감 속도

```
체감 속도 기준:
  100ms   = 즉각 반응 (터치 피드백, 토글)
  300ms   = 빠름 (화면 전환, 드롭다운)
  1000ms  = 허용 (데이터 로딩, 검색 결과)
  3000ms+ = 느림 (스켈레톤 필수, 진행 표시기)
  10000ms = 한계 (사용자가 떠날 수 있다)

이미지 최적화:
  포맷: WebP (iOS 14+ / Android 4.0+)
  리사이징: 표시 크기의 2x (Retina 대응)
  지연 로딩: 뷰포트 밖 이미지는 스크롤 접근 시 로드
  placeholder: LQIP(Low Quality Image Placeholder) 또는 dominant color

  썸네일: 50-100KB 이하
  히어로: 200-400KB 이하
  전체 화면: 500KB-1MB 이하

앱 시작 속도:
  Cold Start 목표: 2초 이내
  Warm Start 목표: 1초 이내

  시각적 피드백: 스플래시 → 스켈레톤 → 콘텐츠
  안티패턴: 모든 데이터를 시작 시 로드 (필요한 것만 먼저)

  첫 의미 있는 콘텐츠(First Meaningful Content): 가능한 빨리
    → 캐시된 데이터 먼저 표시 + 백그라운드 갱신

메모리:
  이미지 캐시: 메모리 50-100MB 제한
  리스트: 셀 재사용으로 메모리 일정 유지
  대형 미디어: 화면에서 벗어나면 즉시 해제
```
