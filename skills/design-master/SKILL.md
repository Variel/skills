# Design Guide — 사용자 중심 인터페이스 설계

> **철칙: 디자인은 사용자의 문제를 해결하기 위해 존재한다.**

---

## 0. Working Model — 코드를 쓰기 전에 반드시 작성하라

> **디자인은 문제 해결이다. 동시에 예술이다. 양쪽 모두를 포기하지 말라.**

코드를 쓰기 전에 **세 가지를 작성하라.** 작성하지 않으면 설계를 시작하지 말라.

1. **Visual Thesis** — 분위기, 재질, 에너지를 **한 문장**으로. 이 문장이 이후 모든 결정의 앵커가 된다.
   - 예: "차가운 강철 위에 네온 오렌지가 번지는 긴장감"
   - 예: "햇빛 가득한 린넨 위의 손글씨처럼 부드럽고 따뜻한"
   - 예: "흑백 활자가 종이 위에 찍히는 에디토리얼의 무게감"

2. **Content Plan** — 화면의 핵심 구조를 4줄 이내로.
   - 랜딩 페이지: hero → support → detail → final CTA
   - 앱: workspace → navigation → context → accent
   - 대시보드: KPIs → charts → tables → filters

3. **Interaction Thesis** — 사용자가 기억할 2-3개의 모션 아이디어.
   - 예: "히어로 진입 시 텍스트가 한 글자씩 찍히는 타이프라이터", "스크롤 시 배경이 시차로 분리", "카드 호버 시 내부 이미지가 미세하게 확대"

### 매번 다르게 만들어라

**같은 폰트, 같은 색상, 같은 패턴으로 수렴하지 말라.** 이전 생성물과 다른 톤·폰트·구조를 의식적으로 선택하라. AI 디자인의 최대 약점은 매번 비슷한 결과를 내는 것이다.

### 주저하지 말라

안전한 선택에 머무르지 말고, visual thesis를 끝까지 밀고 나가라. 틀에서 벗어나 생각하고, 독특한 비전에 완전히 헌신했을 때 진정으로 창조할 수 있는 것을 보여줘라.

## 1. 디자인의 목적

화면 위의 모든 픽셀은 이 질문에 답해야 한다 — "이것이 사용자의 목표 달성에 기여하는가?"

```
필수: 이 요소가 없으면 사용자가 목표를 달성할 수 없다
유용: 이 요소가 있으면 목표 달성이 더 쉬워진다
제거: 둘 다 아니다
```

설계 전에 **전환 맵**을 작성하라 (랜딩 페이지에서는 이것이 카피의 뼈대가 된다):

| 단계 | 질문 | 예시 |
|------|------|------|
| **Before** | 지금 사용자가 겪는 고통은? | "매번 엑셀로 수동 집계" |
| **After** | 제품을 쓴 후의 모습은? | "실시간 대시보드로 즉시 파악" |
| **Bridge** | 제품이 어떻게 연결하는가? | "데이터 연동 → 자동 시각화" |
| **Feeling** | 지배적 감정은? | "안도, 자신감" |

이 네 줄을 작성한 뒤에만 화면 설계를 시작하라.

요소가 다음 네 가지를 지원하지 않으면 제거하라:
1. **탐색** — 원하는 곳으로 이동
2. **이해** — 현재 상태와 맥락 파악
3. **판단** — 선택지 비교, 결정
4. **실행** — 의도를 행동으로

## 2. 사용자의 눈으로 보라

### 제거 테스트 — 주관 판단 대신 사용하라

| 테스트 | 방법 | 실패 시 조치 |
|--------|------|------------|
| **헤드라인 테스트** | 헤드라인만 남기고 나머지를 가린다 | 목적이 전달 안 되면 → 헤드라인 재작성 |
| **이미지 제거 테스트** | 이미지를 제거한다 | 페이지가 여전히 작동하면 → 이미지가 너무 약한 것 |
| **브랜드 테스트** | 내비게이션을 숨긴다 | 브랜드가 사라지면 → 위계가 너무 약한 것 |
| **카피 삭감 테스트** | 카피의 30%를 지운다 | 페이지가 나아지면 → 아직 너무 많은 것. 계속 줄여라 |
| **카드 제거 테스트** | 모든 카드 테두리/그림자를 제거한다 | 의미가 보존되면 → 카드가 불필요한 것. 제거하라 |
| **장식 제거 테스트** | 모든 장식적 그림자를 제거한다 | 프리미엄하지 않게 느껴지면 → 구조가 약한 것 |

### 사용자는 읽지 않고 훑는다

**기본값**: 첫 1-2개 요소가 화면의 역할을 설명해야 한다.
- 헤드라인은 부가 설명 없이도 이해 가능해야 한다
- 사용자가 추측해야 한다면, 재설계하라

### 인지 부하를 줄여라

**Recognition over Recall.** 시스템이 사용자 대신 기억하게 하라.

**기본값**: 핵심 정보를 먼저, 고급 컨트롤은 맥락에 따라 드러낸다. 세부 뷰는 요청 시에만 펼쳐라.
- 모든 기능을 한 번에 보여주는 것은 친절이 아니라 폭력이다
- 관련 행동을 필요한 시점에 노출하라

### 모든 행동에 피드백을 제공하라

상호작용 후의 침묵은 용납할 수 없다.

| 상황 | 반드시 제공할 피드백 |
|------|---------------------|
| 버튼 클릭 | 즉각적인 시각적 반응 |
| 데이터 로딩 | 로딩 상태 표시 |
| 작업 성공 | 명확한 확인 메시지 |
| 오류 발생 | 구체적 오류 설명 + 해결 방법 |
| 데이터 없음 | 빈 상태 안내 + 다음 행동 제안 |
| 권한 부족 | 이유 설명 + 대안 안내 |

### 카피 규칙 — 마케팅과 유틸리티를 구분하라

| 맥락 | 카피 스타일 | 기본값 |
|------|-----------|--------|
| **랜딩 페이지** | 마케팅 카피 | 감정, 약속, 전환. 헤드라인이 의미를 전달 |
| **앱/대시보드** | 유틸리티 카피 | 상태, 행동, 범위. 운영자가 즉시 이해 |

**유틸리티 카피 기본값** (앱/대시보드에서):
- 섹션 헤딩은 영역이나 행동을 말한다. 좋은 예: "선택된 KPI", "플랜 현황", "최근 동기화"
- 히어로 섹션을 넣지 마라. 작업 표면(KPI, 차트, 필터, 테이블)부터 시작. 예외: 사용자가 명시적으로 요청할 때만.
- 홈페이지 히어로나 광고에 들어갈 법한 문장이면 다시 써라. "Unleash your potential" → "이번 주 성과"
- 검증: 운영자가 헤딩과 숫자만 훑어도 페이지를 즉시 이해할 수 있는가?

**마케팅 카피 기본값** (랜딩 페이지에서):
- 헤드라인이 의미를 전달한다. 보조 카피는 한 문장.
- 프롬프트 언어나 디자인 해설을 UI에 넣지 마라.
- 섹션마다 하나의 책임: 설명, 증명, 심화, 또는 전환.
- 카피의 30%를 지워서 나아지면, 계속 지워라.

## 3. 원칙 + 기본값

각 원칙에는 **기본값**(즉시 실행 가능한 규칙)과 **예외 조건**(이탈이 허용되는 상황)이 있다.

### 3.1 목적 우선(Purpose First)

**원칙**: 화면당 하나의 지배적 목적.

**기본값**: 화면당 하나의 주된 행동. 하나의 지배적 사용자 질문에 답하라.
**예외**: 대시보드 개요 화면은 여러 요약을 포함할 수 있지만, 그래도 하나의 지배적 영역이 있어야 한다.
**검증**: "이 화면은 ___를 위한 것이다"를 한 문장으로 말할 수 없으면 분리하라.

### 3.2 지배 영역 규칙(Dominant Region Rule)

**원칙**: 시각적 무게는 중요도를 반영해야 한다.

**기본값**: 지배 영역이 화면의 60-70%를 차지한다. 동일한 무게의 레이아웃 금지.
**예외**: 비교 화면(A vs B)은 의도적으로 동일한 무게를 사용할 수 있다.
**검증**: 한 영역을 가려도 나머지가 독립적으로 읽히면 → 위계가 없는 것이다. 지배 영역을 강화하라.

### 3.3 카드 사용 규칙

**원칙**: 카드는 시각적 컨테이너가 아니라 상호작용의 단위다.

**기본값**: 카드 없이 섹션, 칼럼, 여백으로 구분한다. 카드를 쓰지 않는 것이 기본이다.
**예외**: 카드 자체가 클릭/드래그/선택의 대상일 때만 카드를 쓴다 (칸반 보드, 상품 목록, 갤러리).
**검증**: 카드 테두리와 그림자를 제거해도 의미가 보존되면 → 카드가 불필요한 것이다. 제거하라.

### 3.4 행동 위계(Action Hierarchy)

**원칙**: 주된 행동이 시각적으로 가장 돋보여야 한다.

| 수준 | 기본값 |
|------|--------|
| **주된 행동** | 화면당 하나. 채워진(filled) 버튼, 가장 큰 크기, 가장 눈에 띄는 색상 |
| **보조 행동** | 아웃라인 버튼 또는 텍스트 버튼. 시각적으로 축소 |
| **파괴적 행동** | 빨간색 계열. 실행 전 확인 필수 |
| **드문 행동** | 오버플로 메뉴(⋯)에 배치 |

### 3.5 구조적 일관성(Structural Consistency)

**원칙**: 비슷한 문제는 비슷한 방식으로 해결하라.

**기본값**: 간격은 4의 배수(4, 8, 12, 16, 24, 32, 48px). 색상은 시맨틱 토큰. 내비게이션 패턴은 앱 전체에서 동일.
**예외**: 의도적인 비대칭은 허용하되, 간격 척도는 유지하라.
**검증**: 두 개의 유사한 화면을 나란히 놓았을 때 같은 시스템에서 왔다고 느껴지는가?

### 3.6 확장성(Scalability)

**원칙**: 데이터가 10배 늘어나도 구조가 깨지지 않게 설계하라.

**기본값**: 리스트는 가상 스크롤 또는 페이지네이션. 빈 상태, 로딩 상태, 에러 상태를 모든 데이터 영역에 설계하라.
**예외**: 데이터가 확정적으로 적은 경우(설정 화면의 3-5개 옵션)는 고정 레이아웃 허용.
**검증**: 항목 수를 5 → 50 → 500으로 상상해보라. 어느 지점에서 깨지는가?

## 4. 설계 방법론

### 밀도를 의도적으로 선택하라

**기본값: Medium.** 명시적 이유 없이 다른 모드를 쓰지 마라.

| 모드 | 기본 간격 | 용도 | 전환 조건 |
|------|----------|------|----------|
| **Compact** | 4-8px | 트레이딩, 모니터링, 데이터 테이블 | 한 화면에 20+개 항목이 보여야 할 때 |
| **Medium** | 12-16px | 일반적 기본값 | 이유 없으면 이것 |
| **Airy** | 24-48px | 온보딩, 설정, 랜딩 페이지 | 초보 사용자 대상이거나 감성 전달이 목적일 때 |

하나의 화면 안에서 밀도 모드를 혼합하지 마라.

### 공간을 구조로 사용하라

**기본값**: 장식용 구분선 대신 여백으로 구분한다. 구분선은 여백만으로 충분히 구분되지 않을 때만.

- 하나의 화면에 **하나의 지배적 축**을 유지하라
- 관련 요소 사이 간격 < 비관련 요소 사이 간격 (근접성의 법칙)

### 타이포그래피 제한

**기본값**: 두 서체 이내. 하나의 강조색.
**예외**: 세 번째 서체는 명확한 역할이 있을 때만 (예: 코드용 모노스페이스).

### 엔티티 무결성을 지켜라

어떤 엔티티를 표현하든: **이름**은 눈에 띄게, **상태**는 명확하게, 핵심 **메타데이터**를 표시하고, **행동**을 명백하게 드러내라.

### 반응형 설계

| 디바이스 | 기본값 |
|---------|--------|
| **모바일** | 단일 칼럼. 보조 패널은 시트/드로어. 수평 스크롤 금지. 터치 타겟 44px+ |
| **태블릿** | 사이드바 접힘 가능. 2칼럼까지 허용 |
| **데스크톱** | 다중 영역 허용. 최대 콘텐츠 폭 1280-1440px. 그 이상은 여백 |

## 5. 플랫폼별 가이드 — 반드시 참조하라

### 언제 읽는가 (Hard Gate)

**디자인 작업을 시작하기 전에, 대상 플랫폼의 개요 문서를 반드시 먼저 읽어라.** 이 단계를 건너뛰면 안 된다.

```
사용자 요청 수신
  ↓
★ Working Model 작성 (섹션 0의 세 가지)
  ↓
대상 플랫폼 판별 (웹 앱 / 모바일 앱 / 랜딩 페이지 / 슬라이드)
  ↓
★ 해당 플랫폼 개요 문서를 읽는다 ← 이 단계가 필수
  ↓
개요 문서의 워크플로우에 따라 세부 문서를 순서대로 참조
  ↓
설계 시작
  ↓
visual-and-verification.md로 마감 검증
```

복합 요청(예: "웹 앱 + 랜딩 페이지")이면 관련 개요 문서를 모두 읽되, **화면마다 지배적 패러다임 하나를 명확히 선택하라.**

### 플랫폼별 진입점

| 플랫폼 | 개요 문서 | 하위 세부 문서 |
|--------|----------|--------------|
| **웹 앱** | `web-app.md` | → paradigms, components, practical |
| **모바일 앱** | `mobile-app.md` | → paradigms, components, practical |
| **랜딩 페이지** | `landing-page.md` | → paradigms, sections, practical |
| **슬라이드** | `slides.md` | (단일 문서에 통합) |

## 6. Hard Rules + 실패 장면

### Hard Rules — 절대 위반 금지

- 카드를 기본으로 쓰지 마라. 카드는 상호작용 단위일 때만.
- 히어로에 카드를 넣지 마라.
- full bleed가 요구될 때 center-column 히어로를 쓰지 마라.
- 섹션당 하나의 지배적 아이디어만. 여러 작은 UI 장치로 설명하려 하지 마라.
- 브랜드 페이지에서 헤드라인이 브랜드명보다 커서는 안 된다.
- 채울 말이 없으면 카피를 지어내지 마라 (filler copy 금지).
- 두 서체를 넘지 마라. 예외: 명확한 역할이 있을 때.
- 강조색은 하나. 예외: 제품에 이미 확립된 색상 체계가 있을 때.
- 앱 UI에 마케팅 카피를 넣지 마라. 대시보드에 "Unleash your potential" 금지.

### 이 장면이 보이면 즉시 수정하라

| 실패 장면 | 수정 방향 |
|----------|----------|
| 대시보드 첫 화면이 동일 크기 카드 6개의 그리드 | 하나를 크게 만들거나 카드를 제거하고 섹션으로 |
| 아름다운 히어로 이미지 위에 브랜드명이 본문과 같은 크기 | 브랜드를 가장 큰 텍스트로 |
| 사이드바와 메인 영역이 50:50으로 분할 | 25:75 또는 사이드바를 접을 수 있게 |
| 섹션마다 "더 나은 미래를 만들어요" 같은 비슷한 감성 카피 반복 | 각 섹션에 고유한 역할(설명/증명/심화/전환) 부여 |
| 앱 UI 전체가 카드를 쌓아놓은 것처럼 보임 | 카드 테두리와 그림자 제거, 레이아웃으로 구분 |
| 히어로에 stat strip + logo cloud + pill 배지가 동시에 | 하나만 남기고 나머지 제거 |
| 이미지 위의 텍스트가 읽기 어려움 | 텍스트를 이미지의 차분한 영역에 배치하거나 오버레이 추가 |
| 캐러셀이 있지만 내러티브 목적이 없음 | 캐러셀 제거. 정적 레이아웃 또는 스크롤 섹션으로 |
| 강한 헤드라인은 있지만 다음 행동이 없음 | CTA를 추가하라 |

### AI 디자인 클리셰 — 이것을 발견하면 다시 생각하라

| 클리셰 | 대안 |
|--------|------|
| Inter, Roboto, Arial 반복 | 매번 다른 독특한 폰트 |
| 보라색 그라디언트 + 흰 배경 | 대담한 색상 선택, 텍스처, 패턴 |
| 모든 요소 둥근 모서리 16px | 맥락에 맞는 radius. 날카로운 것도 가능 |
| 동일한 hover opacity 변화 | scale, 색상, 그림자, 변형 |
| 매 생성마다 같은 폰트/색상 수렴 | 이전 결과와 명시적으로 다르게 |
| 단색 배경에 안주 | 텍스처, 그라디언트 메시, 노이즈, 패턴 |

## 7. 검증 — 제거 테스트로 확인하라

**`visual-and-verification.md`를 반드시 참조하라.** 아래는 빠른 검증.

### 제거 테스트

- [ ] 헤드라인만 남겼을 때 페이지 목적이 전달되는가? → 아니면 헤드라인 재작성
- [ ] 가장 큰 영역을 가렸을 때 나머지가 독립적으로 읽히면 → 위계 부재. 지배 영역 강화
- [ ] 이미지를 제거했을 때 페이지가 작동하면 → 이미지가 약한 것. 더 강한 비주얼 앵커
- [ ] 카피의 30%를 지웠을 때 나아지면 → 아직 많은 것. 계속 줄여라
- [ ] 모든 카드 테두리/그림자를 제거했을 때 의미가 보존되면 → 카드 불필요. 제거
- [ ] 모든 장식적 그림자를 제거했을 때도 프리미엄하면 → 통과. 아니면 구조 보강

### 상태 검증

- [ ] 로딩 상태가 있는가?
- [ ] 빈 상태가 있는가? (데이터 없음 + 다음 행동 제안)
- [ ] 에러 상태가 있는가? (구체적 설명 + 해결 방법)
- [ ] 데이터가 10배일 때 깨지지 않는가?

### 미학 검증

- [ ] visual thesis가 끝까지 관철되었는가?
- [ ] AI 클리셰(섹션 6 테이블)에 해당하지 않는가?
- [ ] 이전 생성물과 명시적으로 다른가?
- [ ] 모션이 2-3개 이상 의도적으로 포함되었는가?

상세 검증(타이포, 색상, 모션, 공간 구성, 이미지 위계)은 `visual-and-verification.md`에 정의되어 있다.

## 8. 코드 리뷰 모드 — 기존 UI 코드를 감사하라

디자인 가이드는 새로 만드는 것뿐 아니라 **기존 코드를 검토하는 데에도** 사용된다.

### 언제 사용하는가

- "내 UI 코드 리뷰해줘", "디자인 감사해줘", "접근성 체크해줘"
- "UX 검토해줘", "이 사이트를 베스트 프랙티스와 비교해줘"

### 워크플로우

```
1. 대상 파일/패턴을 확인하라 (사용자가 제공하거나 물어보라)
2. 웹 인터페이스 가이드라인 원본을 가져오라:
   → WebFetch로 아래 URL에서 최신 규칙을 받아온다
   → https://raw.githubusercontent.com/vercel-labs/web-interface-guidelines/main/command.md
3. 대상 파일을 읽어라
4. 이 가이드의 원칙 + Vercel 가이드라인 + visual-and-verification.md 기준으로 검토하라
5. 발견 사항을 `file:line` 포맷으로 간결하게 보고하라
```

### 출력 형식

```
파일:줄번호 — [심각도] 발견 사항
  → 권장 수정 방안
```

심각도: `[필수]` 접근성/기능 장애, `[권장]` UX/디자인 품질, `[제안]` 미학적 개선

### 검토 영역

| 영역 | 검토 항목 |
|------|----------|
| **접근성** | 색상 대비 4.5:1, 포커스 관리, ARIA, 키보드 내비, 터치 타겟 44px+ |
| **의미론** | 시맨틱 HTML, 헤딩 위계, 랜드마크, alt 텍스트 |
| **반응형** | 브레이크포인트 전환, 모바일 터치 타겟, 뷰포트 적응 |
| **성능** | 폰트 로딩, 이미지 최적화, 레이아웃 쉬프트, 애니메이션 GPU |
| **디자인 원칙** | 위계, 일관성, 밀도, 피드백, 빈 상태, 에러 처리 |
| **미학** | AI 클리셰 여부, 폰트 선택, 색상 체계, 공간 구성 |
