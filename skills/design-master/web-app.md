# 웹 애플리케이션 디자인 가이드

> 웹 앱의 설계 사상은 "제품이 사용자에게 무엇을 기대하는가"에 따라 갈린다. 아래 네 가지 패러다임은 각각 다른 전제에서 출발하며, 하나의 제품 안에서도 화면에 따라 섞일 수 있다.

## 패러다임 A: 구조-주도(Structure-Driven) — 화면이 곧 기능 단위

전통적 웹 앱 설계. "화면(page/view)이 사용자의 작업 단위다"라는 전제. 대시보드, 어드민, ERP 등 **기능이 곧 화면**인 제품에 적합.

핵심 사상: **하나의 화면 = 하나의 목적. 화면 안에서 모든 것이 해결된다.**

### 16가지 제품 디자인 법칙

| # | 법칙 | 핵심 규칙 |
|---|------|----------|
| 1 | 목적 우선 | 화면당 하나의 주된 목적 |
| 2 | 지배 영역 | 화면당 하나의 지배적 시각 영역 |
| 3 | 이해 가능성 | 인터페이스는 스스로를 설명해야 한다 |
| 4 | 점진적 공개 | 복잡함을 단계적으로 드러내라 |
| 5 | 인식 > 기억 | 인지 부하를 줄여라 |
| 6 | 시스템 상태 가시성 | Loading/Empty/Error/Success 상태 필수 |
| 7 | 행동 위계 | 주행동 하나, 보조 축소, 파괴적 행동 구분 |
| 8 | 구조적 일관성 | 비슷한 문제 → 비슷한 해결 |
| 9 | 밀도 의도성 | Compact/Medium/Airy 중 하나를 의도적으로 |
| 10 | 공간 논리 | 구조적 레이아웃, 여백으로 구분 |
| 11 | 피드백과 응답 | 모든 행동에 명확한 반응 |
| 12 | 반응형 철학 | 모든 브레이크포인트에서 위계 유지 |
| 13 | 엔티티 무결성 | 이름, 상태, 메타데이터, 행동 표시 |
| 14 | 제약 > 장식 | 기능 없는 요소는 제거 |
| 15 | 확장성 | 성장해도 구조가 유지 |
| 16 | 적응 논리 | 목적에서 구조를 도출 |

### 레이아웃 패턴

```
패턴 A: 사이드바 + 콘텐츠 (대시보드)
┌──────────┬────────────────────────────────┐
│          │                                │
│ 사이드바  │     메인 콘텐츠 영역            │
│  240-280 │      나머지 전체               │
│          │                                │
└──────────┴────────────────────────────────┘

패턴 B: 헤더 + 콘텐츠
┌────────────────────────────────────────────┐
│              헤더 바 (56-64px)              │
├────────────────────────────────────────────┤
│                                            │
│            콘텐츠 영역                      │
│                                            │
└────────────────────────────────────────────┘

패턴 C: 2단 레이아웃
┌─────────────────────┬─────────────┐
│                     │             │
│    메인 (2/3)       │  사이드(1/3)│
│                     │             │
└─────────────────────┴─────────────┘

패턴 D: 카드 그리드
┌──────────┐ ┌──────────┐ ┌──────────┐
│  카드 1  │ │  카드 2  │ │  카드 3  │
└──────────┘ └──────────┘ └──────────┘
```

## 패러다임 B: 명령-주도(Command-Driven) — 사용자가 의도를 직접 말한다

화면 내비게이션 대신 **명령 팔레트(⌘K)와 키보드 중심** 인터랙션. "사용자는 자기가 뭘 하고 싶은지 안다"는 전제.

핵심 사상: **UI를 탐색하지 않고, 의도를 입력한다. 인터페이스는 결과만 보여주면 된다.**

```
참조 제품: Linear, Raycast, Notion, VS Code, Superhuman

작동 원리:
  1. ⌘K로 명령 팔레트를 연다
  2. 자연어/키워드로 의도 입력 ("assign to me", "move to backlog")
  3. 시스템이 맥락 파악해 결과 필터링
  4. Enter로 실행. 화면 전환 없이 작업 완료

설계 원칙:
  - 모든 기능은 명령어로 도달 가능
  - 내비게이션은 선택지이지, 전제가 아니다
  - 키보드 단축키가 1등 시민
  - UI는 "발견 도구"가 아니라 "결과 표시기"

화면 구조:
  ┌────────────────────────────────────────────┐
  │ ⌘K 검색 바 (항상 접근 가능, 전역)            │
  ├──────────┬───────────────────────────────────┤
  │          │                                   │
  │ 최소한의  │   맥락 캔버스                      │
  │ 사이드바  │   (현재 선택된 항목의 상세)          │
  │ (목적지)  │                                   │
  └──────────┴───────────────────────────────────┘
  사이드바: 목적지 5-7개만. 나머지는 ⌘K로.
```

### 명령 팔레트 상세 설계

```
단축키 선택:
  ⌘K        → Superhuman, Linear, Slack (범용, 충돌 적음)
  ⌘+Shift+P → VS Code, Sublime (에디터 전통)
  /         → Slack, Notion (인라인 명령)

  규칙: 같은 단축키로 열고 닫을 수 있어야 한다 (토글)
  규칙: OS 기본 단축키(⌘C, ⌘V, ⌘S)와 절대 충돌하지 말라

팔레트 구조:
  ┌──────────────────────────────────────┐
  │ 🔍 검색어 입력...           [Esc 닫기] │
  ├──────────────────────────────────────┤
  │ 📋 최근 명령                          │
  │   → 이슈 생성          ⌘+N          │
  │   → 백로그로 이동       ⌘+Shift+B    │
  ├──────────────────────────────────────┤
  │ 💡 추천                              │
  │   → 나에게 할당         ⌘+Shift+A    │
  │   → 상태 변경                        │
  └──────────────────────────────────────┘

  너비: 560-640px / 상단에서 20-25% 위치
  최대 결과 수: 화면에 7-10개 (스크롤 허용)
  검색 디바운스: 150-200ms
  결과 하이라이트: 키보드 ↑↓ / Enter 실행 / Esc 닫기
```

**장점**: 숙련자 생산성 극대화, 화면 전환 최소화.
**위험**: 학습 곡선 가파름. 온보딩과 점진적 발견 필수.
**적합**: 개발자 도구, 프로젝트 관리, 이메일, 전문가용 생산성 앱.

## 패러다임 C: 캔버스-주도(Canvas-Driven) — 공간이 곧 인터페이스

고정된 화면 구조 대신 **무한 캔버스 위에서 객체를 직접 조작**. "사용자가 생각을 공간적으로 배열한다"는 전제.

핵심 사상: **화면은 없다. 공간이 있을 뿐이다. 사용자가 구조를 만든다.**

```
참조 제품: Figma, Miro, Obsidian Canvas, tldraw

설계 원칙:
  - 객체 직접 조작(direct manipulation): 드래그, 리사이즈, 연결
  - 줌 아웃 = 개요, 줌 인 = 디테일 (의미적 줌)
  - 도구 팔레트는 선택된 객체에 따라 변화
  - 실행 취소(undo)가 생명줄
  - 격자와 스냅으로 질서 + 자유 배치 허용
  - 수천 개 객체에서도 60fps

화면 구조:
  ┌──────────────────────────────────────────────┐
  │  도구 바 (상단 또는 좌측, 컨텍스트에 따라 변화)  │
  ├──────────────────────────────────────────────┤
  │                                              │
  │          무한 캔버스                           │
  │     (팬/줌, 객체 직접 조작)                    │
  │                         ┌──────────┐         │
  │                         │ 속성 패널  │         │
  │                         └──────────┘         │
  └──────────────────────────────────────────────┘
```

**장점**: 비선형 사고, 협업, 복잡한 관계 시각화.
**위험**: 빈 캔버스 공포. 템플릿과 가이드 필수.
**적합**: 디자인 도구, 화이트보드, 다이어그램, 노트 연결.

## 패러다임 D: 피드-주도(Feed-Driven) — 시간이 곧 구조

화면 구조 대신 **시간순 스트림**이 핵심. "사용자는 '지금 무슨 일이 일어나고 있는가'에 반응한다."

핵심 사상: **사용자가 찾아가지 않는다. 관련 있는 것이 흘러온다.**

```
참조 제품: Twitter/X, Slack, GitHub Activity, Linear Inbox

설계 원칙:
  - 시간이 기본 정렬 축: 최신이 위
  - 각 피드 항목은 자기 완결적: 맥락과 행동 모두 인라인
  - 무한 스크롤 + "새 항목 N개" 알림
  - 읽음/안읽음 상태가 1등 시민
  - 그룹화(오늘/어제)로 시간감 부여
  - 고정(pin)/북마크로 중요 항목 건져올림

화면 구조:
  ┌────────────────────────────────────────────┐
  │  필터 바 (채널, 타입, 날짜 범위)             │
  ├────────────────────────────────────────────┤
  │  ● 새 항목 3개 ↑                            │
  ├────────────────────────────────────────────┤
  │  [아바타] 이름 · 2분 전                      │
  │  피드 항목 콘텐츠                            │
  │  [반응] [답글] [행동]                        │  ← 인라인 액션
  └────────────────────────────────────────────┘
```

**장점**: 실시간성, "놓치지 않았는가"에 대한 안도.
**위험**: 정보 과부하. 필터링/우선순위 알고리즘 필수.
**적합**: 소셜 미디어, 메시징, 활동 로그, 알림 센터, 실시간 모니터링.

---

> **패러다임은 배타적이지 않다.** Linear는 구조-주도 + 명령-주도를 결합. Notion은 구조-주도 + 캔버스-주도를 혼합. 중요한 것은 **각 화면에서 지배적 패러다임이 무엇인지 의식하는 것**.

## 컴포넌트별 상세 설계 패턴

가이드가 "레이아웃 잡고, 색상 정하라"에서 멈추면 실제 화면은 단조로워진다. 아래는 실무에서 반복적으로 등장하는 컴포넌트를, 디자인 시스템들(Atlassian, Carbon, Polaris, Primer, PatternFly)의 공통 규칙으로 정리한 것이다.

### 데이터 테이블

데이터 테이블은 웹 앱에서 가장 복잡한 컴포넌트다. 아이콘 몇 개 나열하는 것과는 차원이 다르다.

```
행 높이 (밀도별):
  Condensed  = 40px   (모니터링, 트레이딩 — 데이터 최우선)
  Regular    = 48px   (기본값 — 대부분의 SaaS)
  Relaxed    = 56px   (관리자 화면 — 행 내 행동 버튼 포함 시)

셀 패딩:
  수평 = 12-16px (좌우)
  수직 = 행 높이에서 폰트 line-height를 뺀 나머지의 절반

열 최소 너비:
  체크박스 열   = 48px (고정)
  아이콘 열     = 56px
  텍스트 열     = 120px 이상
  숫자 열       = 80px 이상 (우측 정렬, 모노스페이스 폰트 필수)
  날짜 열       = 140px 이상
  상태 열       = 100px (배지 포함 시)
  행동 열       = 내용에 따라, 우측 고정(sticky)

헤더:
  높이 = 본문 행과 동일하거나 +4px
  폰트 = 본문보다 1단계 굵게 (예: 500 → 600)
  정렬 방향 표시: ▲▼ 아이콘, 8-12px, 텍스트 우측 4px 간격
  배경: 본문 행보다 약간 어둡게 (Surface-1 또는 Surface-2)

정렬 원칙:
  텍스트 → 좌측 정렬
  숫자 → 우측 정렬 (소수점/자릿수 정렬을 위해)
  상태/배지 → 좌측 정렬
  행동 버튼 → 우측 정렬
  헤더 정렬 = 셀 정렬과 동일

구분선:
  행 구분 = 1px solid, border 색상 사용 (장식 아닌 구조)
  열 구분 = 생략이 기본. 필요 시 1px 극히 연한 색
  줄무늬(zebra) = 짝수 행 Surface-1, 홀수 행 Surface-0 (선택)
```

#### 테이블 상호작용 패턴

```
정렬(Sort):
  클릭 → 오름차순 → 재클릭 → 내림차순 → 재클릭 → 정렬 해제
  다중 정렬: Shift+클릭으로 2차 정렬 추가
  정렬 중인 열 헤더에 시각적 강조 필수

필터링(Filter):
  열 헤더 필터: 드롭다운 or 검색 인풋 (열별 활성화)
  전역 필터 바: 테이블 상단, 칩(chip) 형태로 활성 필터 표시
  활성 필터 개수 배지: "필터 (3)" 처럼 표시
  필터 초기화 버튼: 항상 보이되, 비활성 시 숨기거나 비활성화

페이지네이션 vs 무한 스크롤:
  ┌─────────────────┬──────────────────────────────┐
  │ 페이지네이션     │ 무한 스크롤                    │
  ├─────────────────┼──────────────────────────────┤
  │ 데이터 >500행   │ 피드/타임라인 형태              │
  │ 특정 위치 재방문 │ 순차적 탐색 (SNS, 로그)        │
  │ URL 공유 필요   │ 빠른 훑어보기 위주              │
  │ 서버 부하 관리  │ 데이터 <200행 또는 가상 스크롤   │
  └─────────────────┴──────────────────────────────┘

  페이지네이션 바: 테이블 하단, 높이 48-56px
  표시 항목: "1-25 of 342" + 이전/다음 + 페이지 크기 선택(25/50/100)

인라인 편집:
  활성화: 셀 클릭 또는 더블 클릭 (제품마다 다름 — 일관되게 선택하라)
  호버 시: 연필 아이콘 또는 셀 배경 미세 변화로 편집 가능 힌트
  편집 모드: 셀이 인풋 필드로 전환, 기존 값 선택(selected) 상태
  저장: Enter 또는 셀 바깥 클릭 → 자동 저장
  취소: Esc → 원래 값 복원
  피드백: 저장 성공 시 셀 배경 초록 플래시(300ms) → 원래로

대량 선택(Bulk Actions):
  체크박스 열 최상단: 전체 선택/해제 토글
  전체 선택 시: "25개 선택됨. 전체 342개를 선택하시겠습니까?" 배너
  선택 시: 테이블 상단에 액션 바 슬라이드 인
  액션 바: 선택 개수 + 주요 행동 3-5개 + "더 보기" 오버플로
  파괴적 행동: 빨간 버튼 + 확인 모달 필수
  체크박스 최소 크기: 데스크톱 24×24px, 터치 44×44px
```

#### 열 리사이즈 & 고정

```
열 리사이즈:
  드래그 핸들: 열 헤더 우측 경계, 커서 col-resize
  더블 클릭: 내용에 맞게 자동 너비 조절
  최소 너비 제한: 열 유형별 최소 너비 이하로 줄이지 말라

열 고정(Sticky/Pinned):
  첫 번째 열(식별자): 수평 스크롤 시 좌측 고정
  행동 열: 우측 고정
  고정 열 경계: 그림자(box-shadow) 4px로 깊이감 표현
  고정 열 배경: 불투명(opacity: 1) — 스크롤 시 콘텐츠가 비치지 않게
```

### 폼 설계

```
필드 높이 (밀도별):
  Compact = 32px     Medium = 40px     Large = 48px

레이블:
  위치: 필드 상단 (수직 폼) 또는 좌측 (수평 폼, 너비 여유 시)
  폰트: 본문보다 1-2px 작게, weight 500-600
  레이블-필드 간격: 4-8px
  필수 표시: 레이블 우측 "*" (색상: destructive)

플레이스홀더:
  색상: muted-foreground (본문 대비 50-60% 밝기)
  내용: 입력 형식 힌트 (예: "YYYY-MM-DD"). 레이블 반복 금지

필드 간 간격: 16-24px (수직), 12-16px (수평 그룹 내)
섹션 간 간격: 32-48px

에러 메시지:
  위치: 필드 바로 아래, 4px 간격
  색상: destructive
  아이콘: ⚠️ 12-16px + 텍스트
  필드 테두리: destructive 색상으로 변경
```

#### 인라인 검증 타이밍 — "빨리 칭찬하고, 늦게 벌하라"

```
규칙 1: 빈 필드에 포커스만 줬을 때 에러를 보여주지 말라 (조급한 검증)
규칙 2: 사용자가 입력 중일 때 에러를 보여주지 말라 (타이핑 중단 기다려라)
규칙 3: 포커스를 떠난 후(blur) 검증하되, 이미 에러 상태인 필드는 즉시 검증

타이밍:
  첫 검증 = blur 이벤트 시점 (필드를 떠날 때)
  에러 상태 필드의 재검증 = keyup + 300ms 디바운스 (입력 중 즉시 피드백)
  성공 표시 = 에러가 해결되는 즉시 초록 체크로 전환

예외: 이메일/비밀번호 강도 같은 실시간 피드백은 keyup + 500ms 디바운스

올바른 흐름:
  빈 필드 → [포커스] → (에러 없음) → [입력] → (에러 없음)
  → [blur] → 검증 → 에러 발견 → 에러 표시
  → [다시 포커스] → [입력] → 300ms 후 재검증 → 올바르면 즉시 성공 표시
```

구조:

  │  ① 기본 정보  ──  ② 상세 설정  ──  ③ 확인   │  진행 표시기
  ├────────────────────────────────────────────┤
  │                                            │
  │  현재 단계 폼 필드                           │
  │                                            │

  완료 단계: 채운 원 + 체크 아이콘 + 연결선 (primary 색)
  현재 단계: 채운 원 + 숫자 (primary 색, 굵게)
  미래 단계: 빈 원 + 숫자 (muted 색)

  - 자동 저장: 단계 전환 시마다 서버에 임시 저장 (이탈 방지)
```

### 빈 상태(Empty States)

빈 화면은 실패가 아니라 **사용자를 행동으로 이끌 기회**다. 빈 상태를 방치하면 사용자는 제품이 고장났다고 느낀다.

```
유형별 설계:

┌─────────────────┬──────────────────────────────────────────────┐
│ 유형             │ 구성 요소                                    │
├─────────────────┼──────────────────────────────────────────────┤
│ 첫 사용          │ 환영 일러스트 + 가치 제안 1줄 + 주 행동 버튼    │
│ (Getting Started)│ "첫 프로젝트를 만들어보세요" → [프로젝트 생성]   │
├─────────────────┼──────────────────────────────────────────────┤
│ 검색 결과 없음    │ 🔍 아이콘 + "결과가 없습니다"                  │
│ (No Results)     │ + 검색어 수정 제안 + 필터 초기화 링크           │
├─────────────────┼──────────────────────────────────────────────┤
│ 필터 결과 없음    │ 필터 아이콘 + "조건에 맞는 항목이 없습니다"     │
│ (No Match)       │ + 활성 필터 표시 + [필터 초기화] 버튼          │
├─────────────────┼──────────────────────────────────────────────┤
│ 에러             │ ⚠️ 아이콘 (destructive 색) + 구체적 에러 설명  │
│ (Error)          │ + [다시 시도] 버튼 + 지원 링크                │
├─────────────────┼──────────────────────────────────────────────┤
│ 권한 없음        │ 🔒 아이콘 + "접근 권한이 필요합니다"            │
│ (No Permission)  │ + 권한 요청 방법 안내 + 관리자 연락 링크        │
├─────────────────┼──────────────────────────────────────────────┤
│ 완료/비어있음     │ ✅ 아이콘 + "모든 할 일을 처리했습니다!"        │
│ (All Done)       │ + 축하 메시지 (희소하게, 남용 금지)             │
└─────────────────┴──────────────────────────────────────────────┘

레이아웃:
  아이콘/일러스트: 48-96px, 회색(기본) 또는 상태별 색상
  제목: 20-24px, weight 600
  설명: 14-16px, muted-foreground
  행동 버튼: 설명 아래 16-24px 간격
  전체: 수직 중앙 정렬, max-width 400px
  풀 페이지: 상단 3xl 여백, 좌우 lg 여백
  컴포넌트 내부(테이블/카드): 수직·수평 중앙, 상하좌우 lg 여백
```

### 로딩 상태

```
스켈레톤 스크린(Skeleton):
  표시 시점: 데이터 요청 후 300ms 이내 응답 없을 때
  (300ms 이내 응답 오면 스켈레톤 건너뛰고 바로 콘텐츠)
  애니메이션: 좌→우 반짝임(shimmer), 1.5-2s 주기
  색상: Surface-2 배경 + Surface-3 반짝임
  형태: 실제 콘텐츠와 동일한 레이아웃 — 텍스트 줄, 아바타 원, 카드 사각형
  line-height: 실제 텍스트와 동일 (레이아웃 쉬프트 방지)

스피너:
  크기: Small 16px, Medium 24px, Large 32px
  용도: 버튼 내부(Small), 영역 부분 로딩(Medium), 페이지 전체(Large)
  버튼 내: 텍스트를 스피너로 교체, 버튼 너비 유지(min-width), 비활성화
  애니메이션: 0.8-1.2s 한 바퀴 회전

프로그래스 바:
  높이: 4px (페이지 상단) 또는 8px (컴포넌트 내)
  색상: primary
  용도: 업로드, 내보내기 등 진행률을 알 수 있는 작업
  불확정(indeterminate): 좌우 왕복 애니메이션

Optimistic UI (낙관적 갱신):
  서버 응답 전에 UI를 먼저 갱신하라
  실패 시: 롤백 + 에러 토스트
  적합: 좋아요, 댓글, 상태 변경 등 실패율 낮은 행동
  부적합: 결제, 삭제 등 되돌리기 어려운 행동
```

### 모달 & 다이얼로그

```
크기 체계:
  Small  = max-width 400px  (확인/알림)
  Medium = max-width 560px  (폼, 설정)
  Large  = max-width 720px  (복잡한 폼, 미리보기)
  XLarge = max-width 960px  (테이블, 멀티패널)
  Full   = 90vw × 90vh      (에디터, 미디어 뷰어)

배경 딤: rgba(0,0,0,0.5) — 클릭 시 닫기 (파괴적 행동 모달은 제외)
등장: fade 150ms + scale(0.95 → 1.0) 200ms ease-out
퇴장: fade 100ms ease-in
포커스 트랩: 모달 내부에 포커스 가두기 필수 (Tab 순환)
Esc: 닫기 (파괴적 행동 모달도 Esc 허용 — 취소와 동일)
스크롤: 모달 내부만 스크롤, 배경 스크롤 잠금(body scroll lock)

확인 다이얼로그 (Confirmation):
  제목: "정말 삭제하시겠습니까?" (행동 명시)
  설명: 결과를 구체적으로 ("이 프로젝트와 관련된 32개 이슈가 함께 삭제됩니다")
  버튼: [취소] + [삭제] (destructive variant)
  고위험 행동: 대상 이름 직접 입력 필수 (GitLab 패턴)
    → 입력 완료 전까지 확인 버튼 비활성화

파괴적 행동 심각도별:
  높음 (되돌릴 수 없음) = 확인 모달 + 이름 입력 + danger 버튼
  중간 (복구 어려움)     = 확인 모달 + danger 버튼
  낮음 (쉽게 되돌림)     = 즉시 실행 + Undo 토스트
```

  열(Lane): 드롭 가능 시 상단 2px primary 라인
  열 간 이동: 카드 중심이 열 영역 안에 들어오면 활성화
  순서 변경: 삽입점에 2px 라인 + 12px 간격 확보 애니메이션

파일 업로드 드롭존:
  기본: 점선 테두리, 아이콘 + "파일을 끌어다 놓으세요"
  드래그오버: 테두리 실선(primary), 배경 primary/10%, 아이콘 확대
  크기: 최소 200×120px
  다중 파일: 업로드 목록 + 진행 바 + 개별 취소 버튼

접근성:
  키보드 대안 필수: Space로 선택, ↑↓로 이동, Enter로 확정
  스크린 리더: aria-grabbed, aria-dropeffect, 실시간 순서 안내
```

## 접근성 상세 가이드

"접근성은 체크리스트가 아니라 설계 원칙이다." 후반에 추가하는 것이 아니라, 첫 와이어프레임부터 고려하라.

```
포커스 관리 (SPA 필수):
  페이지 전환 시: 새 페이지의 <h1> 또는 주 콘텐츠에 포커스 이동
  모달 열림: 모달 첫 포커스 가능 요소로 이동, Tab 트랩
  모달 닫힘: 트리거 요소로 포커스 복원
  동적 콘텐츠 추가: aria-live="polite"로 스크린 리더에 알림
  삭제 행동: 삭제된 항목의 다음/이전 항목으로 포커스 이동

포커스 스타일:
  :focus-visible 사용 (마우스 클릭 시 불필요한 아웃라인 제거)
  아웃라인: 2px solid, primary 색, offset 2px
  고대비 모드 대응: outline-style: solid (dotted/dashed 피함)

키보드 내비게이션:
  모든 인터랙티브 요소: Tab으로 도달 가능
  커스텀 위젯: 방향키(←→↑↓) 내부 이동 + Tab으로 위젯 탈출
  단축키: 최소 ⌘Z(undo), ⌘S(save), Esc(닫기/취소) 지원
  단축키 충돌: OS/브라우저 기본 단축키와 겹치지 않게

ARIA 라이브 리전 결정:
  긴급/차단적 → aria-live="assertive" (에러, 로그아웃 경고)
  비긴급 정보 → aria-live="polite" (성공 메시지, 결과 개수)
  사용하지 말 것 → 캐러셀 자동 전환, 마케팅 배너, 대화상자 열림(aria-haspopup 사용)

색상 대비:
  일반 텍스트: 4.5:1 이상 (WCAG AA)
  대형 텍스트(18px+ 또는 14px+ Bold): 3:1 이상
  UI 컴포넌트(아이콘, 테두리): 3:1 이상
  권장: APCA 기준 Lc 60+ (본문), Lc 45+ (대형 텍스트)

모션:
  prefers-reduced-motion: reduce 미디어 쿼리 존중
  대안: 애니메이션 → 즉시 전환(instant), 또는 지속시간 0ms
  절대 금지: 깜빡임 3회/초 이상 (발작 위험)
```

## 애니메이션 & 트랜지션

```
지속시간 원칙:
  즉각 피드백 (토글, 체크박스)     = 100ms
  작은 전환 (호버, 포커스)         = 150ms
  중간 전환 (드롭다운, 탭)         = 200ms
  큰 전환 (모달, 페이지)           = 300ms
  복잡한 전환 (페이지 전환, 레이아웃) = 400ms
  500ms 이상은 "느리다"고 느낀다 — 피하라

이징 함수:
  ease-out (빠르게 시작, 부드럽게 끝) → 대부분의 등장 애니메이션에 사용
  ease-in (느리게 시작, 빠르게 끝)   → 퇴장 애니메이션
  ease-in-out                      → 위치 이동, 레이아웃 변경
  linear                           → 프로그래스 바, 로딩 스피너

비대칭 타이밍:
  등장 = 300ms / 퇴장 = 200ms (등장이 살짝 더 길어야 자연스럽다)
  모달 등장 = 200ms scale(0.95→1) + fade / 모달 퇴장 = 150ms fade

CSS 커스텀 이징 참고:
  Material: cubic-bezier(0.4, 0, 0.2, 1)  — standard
  Apple:    cubic-bezier(0.25, 0.1, 0.25, 1) — default
  Vercel:   cubic-bezier(0.16, 1, 0.3, 1)  — spring-like
```

## 대시보드 설계 패턴

대시보드는 "모든 것을 한눈에"가 아니라 "가장 중요한 것을 한눈에"다.

```
KPI 카드:
  구조: 레이블(12-14px, muted) + 수치(28-36px, Bold) + 변화(%)(14px, 색상)
  높이: 80-120px
  그리드: 1행에 3-5개 (반응형: 모바일 1-2개)
  수치 변화: ▲ 초록 (긍정), ▼ 빨강 (부정), — 회색 (변화 없음)
  수치 포맷: 1,234 (쉼표) / 12.3K (축약) / $1.2M (단위 포함)

차트 배치:
  1행: KPI 카드 바 (전체 너비)
  2행: 주요 차트 (2/3) + 보조 차트 또는 리스트 (1/3)
  3행: 데이터 테이블 (전체 너비)

  차트당 하나의 인사이트. 차트 상단에 인사이트 헤드라인을 넣어라
  예: "이번 달 매출이 전월 대비 23% 증가했습니다" (차트 위 텍스트)

실시간 업데이트:
  폴링 주기: 15-60초 (용도에 따라)
  업데이트 표시: 수치 변경 시 배경 플래시(200ms) 또는 숫자 카운트업 애니메이션
  마지막 업데이트 시각: 대시보드 우측 상단에 "최종 업데이트: 2분 전"
  연결 끊김: 비커넥티비티 배너 + 마지막 데이터 유지 + 회색조 전환

위젯 레이아웃:
  CSS Grid 사용: grid-template-columns: repeat(12, 1fr)
  위젯 최소 크기: 3칼럼 × 200px 높이
  간격: 16-24px
  사용자 커스텀: 드래그로 위치/크기 조절 (선택적)
```
