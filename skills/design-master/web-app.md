# 웹 애플리케이션 디자인 가이드

> 웹 앱의 설계 사상은 "제품이 사용자에게 무엇을 기대하는가"에 따라 갈린다. 아래 네 가지 패러다임은 각각 다른 전제에서 출발하며, 하나의 제품 안에서도 화면에 따라 섞일 수 있다.

## 패러다임 A: 구조-주도(Structure-Driven) — 화면이 곧 기능 단위

전통적 웹 앱 설계. "화면(page/view)이 사용자의 작업 단위다"라는 전제. 대시보드, 어드민, ERP 등 **기능이 곧 화면**인 제품에 적합.

핵심 사상: **하나의 화면 = 하나의 목적. 화면 안에서 모든 것이 해결된다.**

### 16가지 제품 디자인 법칙

| # | 법칙 | 핵심 규칙 |
|---|------|----------|
| 1 | 목적 우선 | 화면당 하나의 주된 목적 |
| 2 | 지배 영역 | 화면당 하나의 지배적 시각 영역 |
| 3 | 이해 가능성 | 인터페이스는 스스로를 설명해야 한다 |
| 4 | 점진적 공개 | 복잡함을 단계적으로 드러내라 |
| 5 | 인식 > 기억 | 인지 부하를 줄여라 |
| 6 | 시스템 상태 가시성 | Loading/Empty/Error/Success 상태 필수 |
| 7 | 행동 위계 | 주행동 하나, 보조 축소, 파괴적 행동 구분 |
| 8 | 구조적 일관성 | 비슷한 문제 → 비슷한 해결 |
| 9 | 밀도 의도성 | Compact/Medium/Airy 중 하나를 의도적으로 |
| 10 | 공간 논리 | 구조적 레이아웃, 여백으로 구분 |
| 11 | 피드백과 응답 | 모든 행동에 명확한 반응 |
| 12 | 반응형 철학 | 모든 브레이크포인트에서 위계 유지 |
| 13 | 엔티티 무결성 | 이름, 상태, 메타데이터, 행동 표시 |
| 14 | 제약 > 장식 | 기능 없는 요소는 제거 |
| 15 | 확장성 | 성장해도 구조가 유지 |
| 16 | 적응 논리 | 목적에서 구조를 도출 |

## 패러다임 B: 명령-주도(Command-Driven) — 사용자가 의도를 직접 말한다

화면 내비게이션 대신 **명령 팔레트(⌘K)와 키보드 중심** 인터랙션. "사용자는 자기가 뭘 하고 싶은지 안다"는 전제.

핵심 사상: **UI를 탐색하지 않고, 의도를 입력한다. 인터페이스는 결과만 보여주면 된다.**

```
참조 제품: Linear, Raycast, Notion, VS Code, Superhuman

작동 원리:
  1. ⌘K로 명령 팔레트를 연다
  2. 자연어/키워드로 의도 입력 ("assign to me", "move to backlog")
  3. 시스템이 맥락 파악해 결과 필터링
  4. Enter로 실행. 화면 전환 없이 작업 완료

설계 원칙:
  - 모든 기능은 명령어로 도달 가능
  - 내비게이션은 선택지이지, 전제가 아니다
  - 키보드 단축키가 1등 시민
  - UI는 "발견 도구"가 아니라 "결과 표시기"

화면 구조:
  ┌────────────────────────────────────────────┐
  │ ⌘K 검색 바 (항상 접근 가능, 전역)            │
  ├──────────┬───────────────────────────────────┤
  │          │                                   │
  │ 최소한의  │   맥락 캔버스                      │
  │ 사이드바  │   (현재 선택된 항목의 상세)          │
  │ (목적지)  │                                   │
  └──────────┴───────────────────────────────────┘
  사이드바: 목적지 5-7개만. 나머지는 ⌘K로.
```

**장점**: 숙련자 생산성 극대화, 화면 전환 최소화.
**위험**: 학습 곡선 가파름. 온보딩과 점진적 발견 필수.
**적합**: 개발자 도구, 프로젝트 관리, 이메일, 전문가용 생산성 앱.

## 패러다임 C: 캔버스-주도(Canvas-Driven) — 공간이 곧 인터페이스

고정된 화면 구조 대신 **무한 캔버스 위에서 객체를 직접 조작**. "사용자가 생각을 공간적으로 배열한다"는 전제.

핵심 사상: **화면은 없다. 공간이 있을 뿐이다. 사용자가 구조를 만든다.**

```
참조 제품: Figma, Miro, Obsidian Canvas, tldraw

설계 원칙:
  - 객체 직접 조작(direct manipulation): 드래그, 리사이즈, 연결
  - 줌 아웃 = 개요, 줌 인 = 디테일 (의미적 줌)
  - 도구 팔레트는 선택된 객체에 따라 변화
  - 실행 취소(undo)가 생명줄
  - 격자와 스냅으로 질서 + 자유 배치 허용
  - 수천 개 객체에서도 60fps

화면 구조:
  ┌──────────────────────────────────────────────┐
  │  도구 바 (상단 또는 좌측, 컨텍스트에 따라 변화)  │
  ├──────────────────────────────────────────────┤
  │                                              │
  │          무한 캔버스                           │
  │     (팬/줌, 객체 직접 조작)                    │
  │                         ┌──────────┐         │
  │                         │ 속성 패널  │         │
  │                         └──────────┘         │
  └──────────────────────────────────────────────┘
```

**장점**: 비선형 사고, 협업, 복잡한 관계 시각화.
**위험**: 빈 캔버스 공포. 템플릿과 가이드 필수.
**적합**: 디자인 도구, 화이트보드, 다이어그램, 노트 연결.

## 패러다임 D: 피드-주도(Feed-Driven) — 시간이 곧 구조

화면 구조 대신 **시간순 스트림**이 핵심. "사용자는 '지금 무슨 일이 일어나고 있는가'에 반응한다."

핵심 사상: **사용자가 찾아가지 않는다. 관련 있는 것이 흘러온다.**

```
참조 제품: Twitter/X, Slack, GitHub Activity, Linear Inbox

설계 원칙:
  - 시간이 기본 정렬 축: 최신이 위
  - 각 피드 항목은 자기 완결적: 맥락과 행동 모두 인라인
  - 무한 스크롤 + "새 항목 N개" 알림
  - 읽음/안읽음 상태가 1등 시민
  - 그룹화(오늘/어제)로 시간감 부여
  - 고정(pin)/북마크로 중요 항목 건져올림

화면 구조:
  ┌────────────────────────────────────────────┐
  │  필터 바 (채널, 타입, 날짜 범위)             │
  ├────────────────────────────────────────────┤
  │  ● 새 항목 3개 ↑                            │
  ├────────────────────────────────────────────┤
  │  [아바타] 이름 · 2분 전                      │
  │  피드 항목 콘텐츠                            │
  │  [반응] [답글] [행동]                        │  ← 인라인 액션
  └────────────────────────────────────────────┘
```

**장점**: 실시간성, "놓치지 않았는가"에 대한 안도.
**위험**: 정보 과부하. 필터링/우선순위 알고리즘 필수.
**적합**: 소셜 미디어, 메시징, 활동 로그, 알림 센터, 실시간 모니터링.

---

> **패러다임은 배타적이지 않다.** Linear는 구조-주도 + 명령-주도를 결합. Notion은 구조-주도 + 캔버스-주도를 혼합. 중요한 것은 **각 화면에서 지배적 패러다임이 무엇인지 의식하는 것**.

## 실무 수치 — 간격, 타이포, 색상

### 접근법 A: 8px 그리드 시스템 — 구조적 엄밀함

8px 배수 기준. Atlassian, Red Hat, GitHub 등 대규모 디자인 시스템에서 채택.

```
간격 스케일:
  2xs   =  4px   (아이콘 내부 패딩)
  xs    =  8px   (인접 요소 간 최소 간격)
  sm    = 12px   (폼 필드 내부 패딩)
  md    = 16px   (카드 패딩, 리스트 항목 간격)
  lg    = 24px   (섹션 내 그룹 간격)
  xl    = 32px   (섹션 간 간격)
  2xl   = 48px   (페이지 영역 구분)
  3xl   = 64px   (최상위 레이아웃 마진)

타이포그래피 (8px line-height 정렬):
  Caption     12px / line-height 16px (2×8)
  Body-sm     14px / line-height 20px
  Body        16px / line-height 24px (3×8)
  Heading-sm  20px / line-height 28px
  Heading     24px / line-height 32px (4×8)
  Display     32px / line-height 40px (5×8)

버튼 높이:
  Small   = 28px (padding: 4px 12px)
  Medium  = 36px (padding: 8px 16px)
  Large   = 44px (padding: 10px 24px)
```

**적합**: 엔터프라이즈 SaaS, 데이터 중심 대시보드, 대규모 팀 협업 제품.

### 접근법 B: 4px 베이스 + 유연한 스케일

4px 최소 단위. Vercel(Geist), Linear, Tailwind CSS 등 모던 제품.

```
간격 스케일 (Tailwind 호환):
  0.5 = 2px    2   = 8px    5   = 20px
  1   = 4px    3   = 12px   6   = 24px
  1.5 = 6px    4   = 16px   8   = 32px

타이포그래피 (Geist 스타일):
  Caption     11px / line-height 16px / tracking +0.01em / weight 400
  Label       13px / line-height 16px / tracking +0.00em / weight 500
  Body        14px / line-height 20px / tracking  0.00em / weight 400
  Title       16px / line-height 24px / tracking -0.011em/ weight 500
  Heading     24px / line-height 32px / tracking -0.029em/ weight 600
  Display     48px / line-height 48px / tracking -0.040em/ weight 700

버튼:
  Small   = 32px 높이 (padding: 0 12px, font: 13px, radius: 6px)
  Medium  = 36px 높이 (padding: 0 16px, font: 14px, radius: 8px)
  Large   = 44px 높이 (padding: 0 20px, font: 14px, radius: 10px)
  아이콘 전용 = 36×36 (radius: 8px)
```

**적합**: 개발자 도구, 프로젝트 관리 앱, 밀도 높은 모던 SaaS.

### 접근법 C: 콘텐츠 폭 중심 레이아웃

사이드바 없이 콘텐츠 max-width 제한. Notion, Substack, 문서형 앱.

```
콘텐츠 최대 폭:
  읽기 중심   = max-width: 680px  (블로그, 문서, 설정)
  작업 중심   = max-width: 960px  (폼, 리스트, 상세 뷰)
  대시보드    = max-width: 1200px (카드 그리드, 차트)
  전폭(Full)  = 제한 없음        (테이블, 캔버스, IDE)

페이지 마진:
  ≤768px   : padding 16px
  769-1024 : padding 32px
  1025-1440: padding 48px (또는 auto margin + max-width)
  ≥1441    : padding 64px (또는 중앙 정렬)

수직 리듬:
  같은 그룹 내 요소     = 8-12px
  논리적 블록 간        = 24-32px
  페이지 섹션 간        = 48-64px
  페이지 상단 여백      = 32-48px
```

**적합**: 문서 도구, CMS, 블로그, 설정/프로필 페이지.

### 접근법 D: 다크 테마 우선 설계

다크 모드 기본, 라이트 파생. Figma, Discord, Spotify.

```
다크 테마 색상:
  배경:
    Surface-0   = #0A0A0A (페이지 배경)
    Surface-1   = #111111 (카드, 패널)
    Surface-2   = #1A1A1A (호버, 선택)
    Surface-3   = #222222 (인풋, 드롭다운)
  텍스트:
    Primary     = #EDEDED (대비 15.4:1)
    Secondary   = #A0A0A0 (대비 7.1:1)
    Tertiary    = #666666 (대비 3.8:1)
  테두리:
    Subtle      = #1F1F1F
    Default     = #2E2E2E
    Strong      = #444444
  강조:
    Brand       = #3B82F6
    Success     = #22C55E
    Warning     = #F59E0B
    Danger      = #EF4444

라이트 테마 대응:
  Surface-0 → #FFFFFF     Primary text → #111111
  Surface-1 → #F9FAFB     Secondary text → #6B7280
  Surface-2 → #F3F4F6     Border Default → #E5E7EB
```

**적합**: 코드 에디터, 미디어 플랫폼, 데이터 모니터링.

## 반응형 브레이크포인트 전략 비교

```
전략 A — 디바이스 중심 (Bootstrap 전통):
  sm: 576px   md: 768px   lg: 992px   xl: 1200px   2xl: 1400px
  → 직관적, 디바이스 범주 명확 / 768-992px 구간이 어정쩡

전략 B — 콘텐츠 중심 (Tailwind 현대):
  sm: 640px   md: 768px   lg: 1024px  xl: 1280px   2xl: 1536px
  → 콘텐츠 흐름 맞춤, 와이드 모니터 우수 / 디바이스 매핑 덜 직관적

전략 C — 레이아웃 전환 중심 (커스텀):
  compact: ≤640px   (단일 칼럼, 사이드바 숨김)
  medium:  641-1024px (사이드바 접힘 or 오버레이)
  wide:    1025-1440px (사이드바 + 콘텐츠)
  ultra:   ≥1441px  (사이드바 + 콘텐츠 + 보조 패널)
  → 실제 레이아웃 변화에 직결 / 팀 내 합의 필요

사이드바 너비 변화 (전략 C):
  compact  → 0px (hamburger)
  medium   → 64px (아이콘만)
  wide     → 240px (펼침)
  ultra    → 280px + 보조 패널 320px
```
